# 第三章：RAG 架构设计

## 3.1 从朴素 RAG 到高级 RAG

第二章介绍了 RAG 的基本流程，但朴素 RAG（Naive RAG）在实际应用中常常面临以下问题：

- 检索到的文档不相关（噪声太多）
- 关键文档没有被检索到（召回不足）
- 用户问题表述模糊，检索方向偏差
- 检索结果顺序不合理，重要信息被淹没

高级 RAG 架构在三个环节进行优化：**检索前**、**检索中**、**检索后**。

```
    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
    │  检索前优化   │    │   检索优化    │    │  检索后优化   │
    │ Pre-retrieval │───→│  Retrieval   │───→│ Post-retrieval│
    │              │    │              │    │              │
    │ · 查询改写   │    │ · 混合检索   │    │ · 重排序      │
    │ · 查询扩展   │    │ · 语义路由   │    │ · 上下文压缩  │
    │ · HyDE       │    │ · 多路召回   │    │ · 上下文重排  │
    │ · 查询分解   │    │              │    │              │
    └──────────────┘    └──────────────┘    └──────────────┘
```

## 3.2 检索前优化（Pre-retrieval）

检索前优化旨在改善用户查询的质量，让检索方向更精准。

### 3.2.1 查询扩展（Query Expansion）

用户的问题往往简短且模糊，直接检索效果有限。查询扩展通过多种方式丰富查询信息：

**同义词扩展**：

```
原始查询: "如何部署 K8s"
扩展查询: "如何部署 K8s OR Kubernetes 部署 OR 容器编排平台部署"
```

**LLM 扩展**：让 LLM 根据原始查询生成多个变体查询，然后分别检索并合并结果。

```python
# 使用 LLM 生成查询变体
expanded_queries = llm.generate(
    "为以下问题生成3个不同表述的查询：\n"
    "原始问题：如何部署 K8s"
)
# 输出：
# 1. Kubernetes 集群部署步骤
# 2. 容器编排平台 K8s 安装指南
# 3. kubeadm 部署 Kubernetes 教程
```

### 3.2.2 查询分解（Query Decomposition）

复杂问题需要拆解为多个子问题，分别检索再综合。

```
原始问题: "RAG和微调在成本、更新频率和适用场景上的区别是什么？"
    │
    ├── 子问题1: "RAG的成本是多少？"
    ├── 子问题2: "微调的成本是多少？"
    ├── 子问题3: "RAG的知识更新频率如何？"
    ├── 子问题4: "微调后如何更新知识？"
    └── 子问题5: "RAG和微调各自适合什么场景？"
```

每个子问题独立检索，最后将所有检索结果合并为 LLM 上下文。

### 3.2.3 HyDE（Hypothetical Document Embedding）

HyDE 的核心思想是：**让 LLM 先生成一个假设性答案，再用这个假设答案的向量去检索**。

```
用户问题: "RAG 的优势是什么？"
       │
       ▼
  LLM 生成假设答案:
  "RAG 的主要优势包括：1) 无需重新训练模型即可更新知识；
   2) 回答可溯源到具体文档；3) 降低幻觉风险；
   4) 成本低于微调；5) 支持私有数据访问。"
       │
       ▼
  对假设答案做向量化，用于检索
       │
       ▼
  检索到与假设答案语义相似的文档（比直接用问题检索更准）
```

原理：假设答案虽然可能包含错误细节，但其"形状"（讨论的主题、使用的术语）与真实文档高度相似，因此是更好的检索 query。

### 3.2.4 查询路由（Query Routing）

不同类型的问题应该路由到不同的检索策略或数据源。

```
用户问题
   │
   ▼
[路由器/分类器]
   │
   ├── 技术问题 → 技术文档向量库
   ├── 产品问题 → 产品手册向量库
   ├── 人事问题 → HR 知识库
   └── 财务问题 → 财报数据库 (SQL 查询)
```

查询路由可以用 LLM 做零样本分类，也可以用小型分类模型实现。

## 3.3 检索优化（Retrieval）

### 3.3.1 稀疏检索 vs 稠密检索

| 维度 | 稀疏检索（BM25/TF-IDF） | 稠密检索（向量搜索） |
|------|------------------------|---------------------|
| 原理 | 基于词频统计 | 基于语义嵌入向量 |
| 匹配方式 | 关键词精确匹配 | 语义相似度匹配 |
| 优势 | 精确术语匹配、专有名词效果好 | 语义理解、同义词泛化 |
| 劣势 | 无法理解同义词和语义 | 可能忽略精确关键词 |
| 典型算法 | BM25, TF-IDF | HNSW, IVF-PQ |
| 计算方式 | 词项频率 × 逆文档频率 | 向量距离/相似度 |

**关键洞察**：稀疏检索擅长精确匹配（如产品编号 "SKU-2024-A3"），稠密检索擅长语义匹配（如 "部署" 匹配 "安装配置"），两者互补。

### 3.3.2 混合检索（Hybrid Search）

混合检索将稀疏检索和稠密检索结合，取长补短。

```
用户查询
   │
   ├──→ BM25 稀疏检索 → 候选集 A (Top-K)
   │
   └──→ 向量稠密检索 → 候选集 B (Top-K)
              │
              ▼
        [结果融合算法]
              │
              ▼
        最终排序结果
```

### 3.3.3 倒数秩融合（Reciprocal Rank Fusion, RRF）

RRF 是最常用的混合检索结果融合算法。核心公式：

$$\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + r(d)}$$

其中 $d$ 是文档，$r(d)$ 是文档在第 $r$ 个排序列表中的排名位置，$k$ 是平滑常数（通常取 60）。

```
BM25 排序:    [D1, D3, D5, D2, D7]   (排名: D1=1, D3=2, ...)
向量排序:    [D3, D2, D5, D4, D1]   (排名: D3=1, D2=2, ...)

RRF 计算 (k=60):
  D1: 1/(60+1) + 1/(60+5) = 0.0164 + 0.0154 = 0.0318
  D2: 1/(60+4) + 1/(60+2) = 0.0156 + 0.0161 = 0.0317
  D3: 1/(60+2) + 1/(60+1) = 0.0161 + 0.0164 = 0.0325  ← 最高
  D4: 0            + 1/(60+4) = 0.0156                   (仅向量检索命中)
  D5: 1/(60+3) + 1/(60+3) = 0.0323

最终排序: [D3, D5, D1, D2, D4]
```

RRF 的优势：
- 不依赖原始分数的绝对值，只依赖排名
- 天然处理不同检索系统的分数尺度差异
- 简单、鲁棒、无需调参

### 3.3.4 语义路由（Semantic Routing）

根据查询的语义意图，动态选择最合适的检索策略：

```python
# LangChain 语义路由示例
from langchain.utils import SemanticRouter

routes = [
    Route("技术支持", embeddings, ["部署问题", "Bug报告", "性能优化"]),
    Route("产品咨询", embeddings, ["功能介绍", "价格方案", "对比评测"]),
    Route("人事政策", embeddings, ["请假制度", "薪资福利", "晋升流程"]),
]

router = SemanticRouter(routes)
selected_route = router.route("我的服务启动报错了")
# → 路由到 "技术支持"
```

## 3.4 检索后优化（Post-retrieval）

### 3.4.1 重排序（Re-ranking）

向量检索是"双编码器"（Bi-encoder）模式——查询和文档分别编码，再计算相似度。这种方式快但精度有限，因为查询和文档之间没有交叉注意力。

重排序使用"交叉编码器"（Cross-encoder），将查询和文档一起输入模型，进行深度交互，精度显著提升。

```
双编码器（检索阶段）:
  Query → Encoder → Q_vec    ──── 相似度计算 ────  Doc_vec ← Encoder ← Doc
  （快速但粗略）

交叉编码器（重排序阶段）:
  [Query + Doc1] → Cross-Encoder → 相关性分数1
  [Query + Doc2] → Cross-Encoder → 相关性分数2
  [Query + Doc3] → Cross-Encoder → 相关性分数3
  （慢但精准）
```

主流重排序模型：

| 模型 | 提供方 | 特点 | 适用场景 |
|------|--------|------|---------|
| Cohere Rerank | Cohere | API 调用，质量高 | 通用，API 优先 |
| bge-reranker-large | BAAI | 开源，中文优秀 | 中文场景首选 |
| bge-reranker-v2-m3 | BAAI | 多语言，轻量 | 多语言场景 |
| ColBERT | Stanford | 基于交互的延迟交互 | 需要细粒度匹配 |
| Jina Reranker | Jina | 开源，支持长文档 | 长文档场景 |

实践中的两阶段检索范式：
```
第一阶段：向量检索 (Top-100) → 快速召回
第二阶段：交叉编码器重排序 (Top-100 → Top-5) → 精准排序
```

### 3.4.2 上下文压缩与过滤

检索到的文档块可能包含大量无关内容。上下文压缩技术移除噪声，只保留与问题相关的部分。

**提取式压缩**：从文档中提取与问题相关的句子。

**生成式压缩**：用 LLM 将文档压缩为与问题相关的摘要。

**嵌入过滤**：用小模型判断每个文档块是否与问题相关，过滤掉不相关的块。

### 3.4.3 "迷失在中间"问题（Lost in the Middle）

研究表明，LLM 对上下文中间位置的信息关注度显著低于首尾位置。这意味着即使检索到了相关文档，如果关键信息位于上下文中间，模型也可能忽略它。

**重排策略**：将最相关的文档放在上下文的首尾位置，次相关的放在中间。

```
原始排序（按相关性降序）:
  [Doc1: 0.95] [Doc2: 0.89] [Doc3: 0.85] [Doc4: 0.78] [Doc5: 0.72]

优化重排（首尾放最相关）:
  [Doc1: 0.95] [Doc3: 0.85] [Doc5: 0.72] [Doc4: 0.78] [Doc2: 0.89]
   ↑ 首位                                    ↑ 尾位
   高相关                                     高相关
```

## 3.5 高级 RAG 架构模式

### 3.5.1 多模态 RAG（Multimodal RAG）

传统 RAG 只处理文本，但企业数据中大量信息以图片、表格、PDF 版面等形式存在。多模态 RAG 扩展了检索和生成的范围。

```
         PDF 文档
            │
    ┌───────┼───────┐
    │       │       │
  文本    表格    图片
    │       │       │
    ▼       ▼       ▼
 文本块   表格→文本  图片→描述
  向量     向量      向量
    │       │       │
    └───────┼───────┘
            │
            ▼
      统一向量索引
            │
            ▼
        检索 + 生成
```

多模态 RAG 的三种实现路径：

| 路径 | 方法 | 优势 | 劣势 |
|------|------|------|------|
| 嵌入对齐 | 用多模态嵌入模型（CLIP）统一编码 | 端到端，简洁 | 需要多模态嵌入模型 |
| 文本转化 | 将图片/表格转为文字描述，再用文本嵌入 | 兼容现有系统 | 转化过程可能丢失信息 |
| 混合检索 | 文本用文本嵌入，图片用图片嵌入，分别检索 | 保留各模态信息 | 融合策略复杂 |

### 3.5.2 Agent RAG（智能体驱动的 RAG）

传统 RAG 是被动检索——固定流程，不管问题是否需要都执行检索。Agent RAG 赋予系统自主决策能力：

- **决定是否需要检索**：简单常识问题直接回答
- **决定检索什么**：根据问题动态选择数据源和策略
- **决定检索是否充分**：评估检索结果，必要时多轮检索
- **决定如何使用检索结果**：工具调用、推理、交叉验证

```
用户问题
   │
   ▼
┌──────────┐
│  Agent   │ ← 决策中心
│ (LLM)    │
└────┬─────┘
     │
     ├──→ 判断: 需要检索？ ──否──→ 直接回答
     │         │是
     │         ▼
     │    [检索工具1: 向量库] → 结果评估 → 不充分 → [检索工具2: Web搜索]
     │         │                                         │
     │         └─────────────── 充分 ──────────────────────┘
     │                              │
     │                              ▼
     │                    综合推理 + 生成回答
     │
     └──→ 判断: 需要计算？ ──是──→ [计算工具] → 结果 → 整合
```

Agent RAG 的核心优势：
- 自适应：不需要每条查询都走全流程
- 多步推理：复杂问题可以多轮检索和推理
- 工具调用：结合数据库查询、API 调用、代码执行等

### 3.5.3 Graph RAG（知识图谱 RAG）

传统向量检索是基于"文档片段"的扁平检索，缺乏实体之间的关系信息。Graph RAG 将知识图谱与向量检索结合：

```
┌─────────────────────────────────────────┐
│               知识图谱                    │
│                                          │
│  [RAG] ──属于──→ [AI技术] ──包含──→ [检索]  │
│   │                     │                │
│  解决 ──→ [幻觉问题]    包含 ──→ [微调]     │
│   │                     │                │
│  需要 ──→ [向量数据库]  解决 ──→ [领域适配] │
└─────────────────────────────────────────┘
        │
        │ 与向量检索结合
        ▼
  [实体A] 的向量 → 向量检索 → 关联实体 → 图遍历 → 扩展上下文
```

Graph RAG 的优势：
- **关系推理**：通过图遍历发现隐含的实体关系
- **结构化知识**：知识图谱提供明确的三元组关系
- **全局理解**：社区检测算法可以理解文档集的全局主题结构

微软的 GraphRAG 是这一方向的代表性工作，它使用 LLM 从文档中自动提取实体和关系构建知识图谱，再通过社区检测生成全局摘要。

### 3.5.4 Self-RAG（自反思 RAG）

Self-RAG 让 LLM 在生成过程中自我评估是否需要检索、检索结果是否相关、生成内容是否有据可依。

三个核心反思动作：

| 动作 | 触发时机 | 判断标准 |
|------|---------|---------|
| Retrieve | 生成前 | "这个问题需要外部知识吗？" |
| IsREL | 检索后 | "检索到的文档与问题相关吗？" |
| IsSUP | 生成后 | "生成的内容有文档支撑吗？" |

```
用户问题
   │
   ▼
[需要检索?] ──否──→ 直接生成
   │是
   ▼
[检索文档]
   │
   ▼
[文档相关?] ──否──→ 丢弃，重新检索
   │是
   ▼
[基于文档生成]
   │
   ▼
[有据可依?] ──否──→ 重新生成或标注"无依据"
   │是
   ▼
最终输出（附可靠性标记）
```

### 3.5.5 Corrective RAG（CRAG）

CRAG 在检索结果不理想时启动纠正机制：

```
正常路径:
  检索 → [评估器] → 质量合格 → 生成

纠正路径:
  检索 → [评估器] → 质量不合格
                    │
                    ├── 文档轻度偏题 → 查询改写后重新检索
                    ├── 文档严重偏题 → 切换到 Web 搜索
                    └── 无法判断 → 降级为直接生成（标注不确定性）
```

CRAG 的评估器可以是一个轻量级 LLM 或分类模型，判断检索文档与查询的相关程度。

## 3.6 架构模式对比

| 架构模式 | 复杂度 | 检索质量 | 延迟 | 成本 | 适用场景 |
|----------|--------|---------|------|------|---------|
| Naive RAG | 低 | 中 | 低 | 低 | 简单问答、原型验证 |
| Advanced RAG | 中 | 高 | 中 | 中 | 企业级问答、知识库 |
| Multimodal RAG | 中高 | 高 | 中高 | 中高 | PDF 文档、含图表数据 |
| Agent RAG | 高 | 很高 | 高 | 高 | 复杂推理、多步任务 |
| Graph RAG | 高 | 高(关系推理) | 中高 | 高 | 关系密集型、全局分析 |
| Self-RAG | 中高 | 高 | 中 | 中 | 高可靠性需求 |
| CRAG | 中 | 高 | 中 | 中 | 检索质量不稳定时 |

### 选择决策树

```
需要什么类型的 RAG？
   │
   ├── 仅文本问答
   │     ├── 简单快速 → Naive RAG
   │     └── 质量优先 → Advanced RAG
   │
   ├── 含图片/表格/PDF
   │     └── Multimodal RAG
   │
   ├── 需要多步推理
   │     └── Agent RAG
   │
   ├── 需要关系推理/全局摘要
   │     └── Graph RAG
   │
   ├── 需要高可靠性/可验证性
   │     └── Self-RAG
   │
   └── 检索质量不稳定
         └── CRAG
```

## 3.7 本章小结

高级 RAG 架构设计是提升 RAG 系统质量的关键。三个优化环节各有侧重：

**检索前优化**解决"问不对"的问题——查询太模糊、太复杂、方向不对。通过查询扩展、分解、HyDE 和路由，让检索方向更精准。

**检索优化**解决"找不全"的问题——单一检索方式覆盖有限。通过混合检索（BM25 + 向量）和 RRF 融合，大幅提升召回率。

**检索后优化**解决"找不准"的问题——检索结果噪声大、排序不合理。通过交叉编码器重排序、上下文压缩和"迷失在中间"重排，提升最终上下文质量。

高级架构模式（Agent RAG、Graph RAG、Self-RAG、CRAG）则从不同维度突破传统 RAG 的局限：自主决策、关系推理、自我反思、自动纠正。

实际项目中，建议从 Advanced RAG 开始，根据具体痛点逐步引入高级模式，而非一开始就构建最复杂的系统。
