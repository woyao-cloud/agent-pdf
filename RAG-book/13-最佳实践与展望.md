# 13 -- 最佳实践与展望

本章总结 RAG 生产部署的最佳实践，并展望技术发展方向。

## 生产部署最佳实践

### 架构决策

#### 单体 vs 微服务

| 维度 | 单体 | 微服务 |
|------|------|--------|
| 复杂度 | 低 | 高 |
| 部署 | 简单 | 需要 K8s/Docker Compose |
| 扩展 | 整体扩展 | 独立扩展 |
| 故障隔离 | 差 | 好 |
| 适用 | <10K 文档、<100 QPS | >100K 文档、高可用 |

**建议**：从小规模单体开始，当 QPS 超过 100 或文档量超过 100K 时考虑拆分。

#### 同步 vs 异步检索

```python
# 同步：简单直接，适合低延迟场景
result = rag_chain.invoke("查询内容")

# 异步：适合高并发和长耗时查询
result = await rag_chain.ainvoke("查询内容")
# 或使用任务队列
from celery import Celery
app = Celery('rag')

@app.task
def async_rag_query(query: str):
    return rag_chain.invoke(query)
```

#### 缓存策略

```
查询缓存（Query Cache）：
  相同查询 → 直接返回缓存结果
  TTL: 1-24 小时（取决于数据新鲜度要求）
  适用：FAQ 场景、重复查询多

嵌入缓存（Embedding Cache）：
  相同文本 → 直接返回嵌入向量
  避免重复调用嵌入 API
  适用：文档不变的场景

结果缓存（Result Cache）：
  相似查询 → 缓存并返回
  需要语义去重
  适用：查询模式可预测的场景
```

#### 增量索引

```
全量重建：删除旧索引 → 重新嵌入所有文档
  - 简单但开销大
  - 适合：文档量 <10K，更新频率低

增量更新：只嵌入变化的文档
  - 添加：嵌入新文档 → 插入向量库
  - 更新：删除旧向量 → 插入新向量
  - 删除：按 metadata.doc_id 删除
  - 适合：文档量大，频繁更新

变更检测：
  - 基于文件哈希（MD5/SHA256）
  - 基于时间戳
  - 基于 Git 提交
```

#### 数据质量

```
索引前清洗：
  [ ] 去重（内容哈希）
  [ ] 去除空白和特殊字符
  [ ] 统一编码（UTF-8）
  [ ] 修复 PDF 提取错误
  [ ] 处理表格和图片
  [ ] 添加元数据（来源、日期、部门）
  [ ] 语言检测和分离
  [ ] PII 检测和脱敏
```

### 安全与合规

#### 访问控制

```python

from qdrant_client.models import Filter, FieldCondition, MatchValue

results = vectorstore.similarity_search(
    "查询内容",
    k=10,
    filter=Filter(must=[
        # 只检索用户有权限访问的文档
        FieldCondition(key="metadata.department", match=MatchValue(value="engineering")),
        FieldCondition(key="metadata.access_level", match=MatchValue(value="internal")),
    ]),
)
```

#### 数据隐私

```
PII 处理策略：
  1. 索引前脱敏：正则替换姓名、电话、邮箱、身份证号
  2. 嵌入模型本地部署：数据不出内网
  3. 向量数据库加密存储：at-rest encryption
  4. 查询日志脱敏：不记录原始查询，记录查询哈希

合规要求：
  - GDPR: 数据最小化、删除权、数据可携带
  - HIPAA: 医疗数据加密、访问审计
  - SOC2: 安全控制、审计日志
  - 数据主权：确保向量库在合规区域内
```

### 监控与运维

#### 关键指标

```
检索质量：
  - 检索 Recall@K：相关文档被检索到的比例
  - 检索 Precision@K：检索结果中相关文档的比例
  - MRR（Mean Reciprocal Rank）：第一个相关文档的排名倒数的均值
  - 检索延迟 P50/P95/P99

生成质量：
  - 答案相关性：答案是否回应了问题
  - 忠实度：答案是否基于检索到的上下文
  - 幻觉率：答案中无根据陈述的比例

系统指标：
  - 端到端延迟：从查询到响应的总时间
  - 吞吐量：每秒处理查询数
  - 错误率：查询失败比例
  - 成本：每查询平均 token 消耗
```

## 技术展望

### Agentic RAG

Agentic RAG 将 LLM 从被动检索器升级为主动决策者：

```
传统 RAG：用户查询 → 检索 → 生成
Agentic RAG：用户查询 → Agent 决策 → 
  ├─ 是否需要检索？检索什么？如何检索？
  ├─ 检索结果是否足够？需要补充检索？
  ├─ 是否需要调用工具？（SQL 查询、API、计算器）
  ├─ 是否需要多步推理？
  └─ 生成最终答案
```

**DeepAgents 中的 Agentic RAG**：

```python
from deepagents import create_deep_agent, SubAgent

research_agent: SubAgent = {
    "name": "researcher",
    "description": "深度研究代理，搜索和分析信息",
    "system_prompt": """你是一个研究代理。使用以下策略：
    1. 先检索相关文档
    2. 如果结果不够，使用 web_search 补充
    3. 如果需要数据，使用 SQL 查询
    4. 综合所有信息，返回结构化报告""",
    "tools": [search_tool, sql_tool, web_search_tool],
}

agent = create_deep_agent(
    subagents=[research_agent],
    system_prompt="你是一个知识库助手。使用研究代理进行深度检索。",
)
```

**Agentic RAG 的关键特征**：
- **自适应检索**：Agent 自行决定是否需要检索、检索几次
- **工具调用**：集成 SQL、API、计算器等外部工具
- **多步推理**：Chain-of-Thought + ReAct 模式
- **自我评估**：检索后评估结果质量，决定是否继续

### 多模态 RAG

```
文本 RAG：文档 → 嵌入 → 向量搜索 → 生成
  ↓
多模态 RAG：
  图像 → CLIP/GPT-4V 嵌入 → 多模态向量搜索 → 生成
  表格 → 结构化提取 → 语义嵌入 → 生成
  PDF（含图表）→ LayoutLM → 分区嵌入 → 生成
  视频 → 关键帧提取 → 帧嵌入 → 生成
  音频 → Whisper 转写 → 文本嵌入 → 生成
```

**关键技术**：
- **CLIP**：图像-文本联合嵌入空间
- **GPT-4V / Claude Vision**：直接理解图像内容
- **LayoutLM / Donut**：文档布局理解
- **Whisper**：音频转写

### Graph RAG

知识图谱 + 向量检索的融合：

```
传统向量 RAG：查询 → 向量搜索 → 文本块 → 生成
  ↓
Graph RAG：查询 → 向量搜索 → 实体识别 → 
  图谱遍历（关系、社区）→ 结构化上下文 → 生成
```

**Microsoft GraphRAG 方法**：
1. 从文档中提取实体和关系
2. 构建知识图谱
3. 检测社区结构
4. 为每个社区生成摘要
5. 查询时：局部查询用向量搜索，全局查询用社区摘要

**优势**：
- 多跳推理（"A 的经理的团队有哪些项目？"）
- 全局性理解（"文档集的主要主题是什么？"）
- 实体关系挖掘

### Corrective RAG (CRAG)

自我纠错的 RAG 流程：

```
用户查询
  ↓
检索文档
  ↓
评估检索质量（LLM 评判）
  ├─ 相关度 ≥ 阈值 → 直接生成
  ├─ 部分相关 → 细化查询 → 重新检索 → 生成
  └─ 不相关 → Web 搜索补充 → 生成
```

**关键组件**：
- **检索评估器**：判断检索结果的相关性
- **查询细化器**：改写查询提高召回
- **知识精炼器**：从检索文档中提取关键信息
- **备用检索**：Web 搜索作为兜底

### 自适应 RAG (Self-RAG)

LLM 自行决定何时检索、何时不检索：

```
用户查询
  ↓
LLM 判断：是否需要检索？
  ├─ 不需要 → 直接生成（有足够知识）
  └─ 需要 → 检索 → 评估：相关吗？
       ├─ 相关 → 使用检索结果生成
       └─ 不相关 → 丢弃检索结果，直接生成
```

**核心创新**：引入特殊 token：
- `[Retrieve]`：需要检索
- `[ISREL]`：检索结果相关
- `[ISREL]`：检索结果不相关
- `[ISREL]`：不需要检索

### 小模型 + RAG

```
传统 RAG：大模型（GPT-4, Claude）+ 向量检索
  ↓
经济 RAG：小模型（Phi-3, Llama-3.1-8B）+ 向量检索
  ↓
极致经济 RAG：小模型 + 高质量嵌入 + 严格重排序 + 精简上下文
```

**成本对比**：
| 方案 | 每查询成本 | 延迟 | 质量 |
|------|-----------|------|------|
| GPT-4 + RAG | ~$0.05 | 3-5s | 最高 |
| Claude + RAG | ~$0.03 | 3-5s | 高 |
| Llama-3.1-8B + RAG | ~$0.001 | 1-2s | 中高 |
| Phi-3-mini + RAG | ~$0.0003 | <1s | 中 |

**优化技巧**：
- 用大模型做重排序，小模型做生成
- 用小模型做分类路由，大模型做复杂推理
- LoRA 微调小模型在领域数据上
- 量化（GPTQ、AWQ、bitsandbytes）降低推理成本

## 技术雷达

| 技术 | 状态 | 建议 |
|------|------|------|
| 向量搜索 + BM25 混合检索 | 采纳 | 生产必备 |

```python

框架: Haystack (管道序列化、可观测性)

监控: LangSmith + Prometheus + Grafana
```

```python

重排序: BGE-reranker (本地部署)

```

## 总结

RAG 技术正在快速演进，从最初的"检索 + 生成"简单组合，发展到今天的多策略、多模态、智能体驱动的复杂系统。核心趋势：

1. **从静态到动态**：Agentic RAG 让系统自主决定检索策略
2. **从文本到多模态**：图像、表格、视频的检索和生成
3. **从简单到混合**：BM25 + 向量 + 知识图谱的融合
4. **从通用到领域**：微调嵌入和模型提升领域准确性
5. **从昂贵到经济**：小模型 + 高质量检索 = 足够好的结果

掌握 RAG 的开发者，需要同时理解：向量数据库原理、嵌入模型选择、检索策略优化、LLM 提示工程和生产部署实践。本书覆盖的这些知识，正是构建生产级 RAG 系统的基础。
