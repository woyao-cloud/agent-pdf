# 第15章 权限与角色控制

## 15.1 概念与语法

### 15.1.1 基于角色的访问控制（RBAC）

RBAC（Role-Based Access Control）是NIST标准RBAC模型（INCITS 359-2004）定义的访问控制范式。其核心思想是：**权限不直接分配给用户，而是分配给角色，用户通过关联角色获得权限**。这种间接层带来了巨大的管理便利——当权限变更时，只需修改角色定义，而非逐一修改每个用户。

```
经典RBAC三元组：

┌────────┐     N:M     ┌────────┐     N:M     ┌────────┐
│  User  │◄──────────►│  Role  │◄──────────►│  Perm  │
│ 用户   │  用户-角色   │  角色  │  角色-权限   │  权限  │
└────────┘  关联表     └────────┘  关联表     └────────┘

示例：
User: alice ──► Role: admin ──► Perm: user:delete
                              ──► Perm: article:publish
                              ──► Perm: system:config

User: bob   ──► Role: editor ──► Perm: article:publish
                              ──► Perm: article:edit

User: carol ──► Role: viewer ──► Perm: article:read
```

**RBAC的层次模型**：

```
RBAC0: 基本层 - 用户-角色-权限的简单映射
RBAC1: 角色继承 - 角色可以继承其他角色的权限
RBAC2: 约束 - 互斥角色、基数约束（如一个用户最多N个角色）
RBAC3: 组合 - RBAC1 + RBAC2
```

**权限的命名约定**：

```python
# 推荐使用 resource:action 格式
permissions = {
    "user:create",     # 创建用户
    "user:read",       # 查看用户
    "user:update",     # 更新用户
    "user:delete",     # 删除用户
    "article:create",  # 创建文章
    "article:read",    # 查看文章
    "article:update",  # 更新文章
    "article:delete",  # 删除文章
    "article:publish", # 发布文章
    "system:config",   # 系统配置
}
```

这种命名约定有以下优势：
- 前缀标识资源类型，便于按资源分组检查
- 后缀标识操作类型，与CRUD语义对齐
- 支持通配符匹配（如`article:*`表示所有文章操作）

### 15.1.2 权限依赖

FastAPI中实现权限控制的核心手段是**权限依赖**（Permission Dependencies）。通过依赖注入系统，将权限检查声明为路由参数，框架自动在请求处理前执行验证。

```python
from fastapi import Depends, HTTPException, status

# 基础权限依赖
def require_permission(permission: str):
    """创建一个要求特定权限的依赖项"""
    async def permission_checker(
        current_user: TokenData = Depends(get_current_user),
    ) -> TokenData:
        user_permissions = get_user_permissions(current_user.user_id)
        if permission not in user_permissions:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"权限不足：需要 '{permission}'",
            )
        return current_user
    return permission_checker


# 使用
@app.delete("/api/users/{user_id}")
async def delete_user(
    user_id: str,
    current_user: TokenData = Depends(require_permission("user:delete")),
):
    # 只有拥有 user:delete 权限的用户才能到达这里
    await user_service.delete(user_id)
```

**权限依赖的嵌套组合**：

```python
# 多权限检查（AND语义 - 需要所有权限）
def require_all_permissions(*permissions: str):
    """要求所有指定权限"""
    async def checker(
        current_user: TokenData = Depends(get_current_user),
    ) -> TokenData:
        user_perms = get_user_permissions(current_user.user_id)
        missing = [p for p in permissions if p not in user_perms]
        if missing:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"缺少权限: {', '.join(missing)}",
            )
        return current_user
    return checker


# 多权限检查（OR语义 - 只需任一权限）
def require_any_permission(*permissions: str):
    """要求任一指定权限"""
    async def checker(
        current_user: TokenData = Depends(get_current_user),
    ) -> TokenData:
        user_perms = get_user_permissions(current_user.user_id)
        if not any(p in user_perms for p in permissions):
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"需要以下任一权限: {', '.join(permissions)}",
            )
        return current_user
    return checker


# 使用
@app.post("/api/articles/{article_id}/publish")
async def publish_article(
    article_id: str,
    current_user: TokenData = Depends(
        require_any_permission("article:publish", "article:update")
    ),
):
    # 拥有 publish 或 update 权限均可发布
    pass
```

### 15.1.3 权限装饰器

虽然FastAPI推荐使用依赖注入，但装饰器风格在某些场景下更简洁。可以通过装饰器封装权限检查逻辑，然后在路由函数上声明。

```python
from functools import wraps
from typing import Callable

def permission_required(permission: str):
    """
    权限装饰器 - 在路由函数上声明所需权限
    注意：此装饰器需要与FastAPI依赖注入配合使用
    """
    def decorator(func: Callable):
        # 将权限信息附加到函数上，供依赖注入读取
        if not hasattr(func, "_required_permissions"):
            func._required_permissions = []
        func._required_permissions.append(permission)

        @wraps(func)
        async def wrapper(*args, **kwargs):
            return await func(*args, **kwargs)
        # 保留权限元数据
        wrapper._required_permissions = func._required_permissions
        return wrapper
    return decorator


# 更实用的方式：使用类作为依赖
from fastapi import Request

class PermissionChecker:
    """权限检查器 - 类依赖方式"""

    def __init__(self, permission: str):
        self.permission = permission

    async def __call__(
        self,
        request: Request,
        current_user: TokenData = Depends(get_current_user),
    ) -> TokenData:
        user_perms = get_user_permissions(current_user.user_id)
        if self.permission not in user_perms:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"权限不足：需要 '{self.permission}'",
            )
        return current_user


# 使用
@app.delete("/api/users/{user_id}")
async def delete_user(
    user_id: str,
    current_user: TokenData = Depends(PermissionChecker("user:delete")),
):
    await user_service.delete(user_id)
```

### 15.1.4 动态权限检查

动态权限检查是指权限判定需要运行时信息（如请求参数、资源所有者）才能完成的检查。最典型的场景是**资源级权限**——"用户只能编辑自己的文章"。

```python
from fastapi import Path

async def can_modify_article(
    article_id: str = Path(...),
    current_user: TokenData = Depends(get_current_user),
) -> TokenData:
    """
    动态权限检查：用户可以修改自己的文章，或admin可以修改任何文章
    """
    article = await article_service.get(article_id)
    if article is None:
        raise HTTPException(status_code=404, detail="文章不存在")

    # 检查是否是文章作者
    if article.author_id != current_user.user_id:
        # 检查是否有admin权限
        user_perms = get_user_permissions(current_user.user_id)
        if "article:update:others" not in user_perms:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail="只能修改自己的文章",
            )

    return current_user


@app.put("/api/articles/{article_id}")
async def update_article(
    article_id: str,
    current_user: TokenData = Depends(can_modify_article),
):
    # 到达这里的一定是有权修改的用户
    pass
```

## 15.2 原理与机制

### 15.2.1 依赖注入实现权限链

FastAPI的权限控制完全构建在依赖注入系统之上。理解权限链的执行原理，需要深入FastAPI的依赖解析机制。

**依赖图构建**：

```
路由函数: delete_user(user_id, current_user=Depends(require_permission("user:delete")))
    │
    └── require_permission("user:delete")
            │
            └── get_current_user(token=Depends(oauth2_scheme))
                    │
                    └── oauth2_scheme(request)  ← 从HTTP请求提取Token

依赖解析顺序（深度优先）：
1. oauth2_scheme(request) → token: str
2. get_current_user(token) → TokenData
3. require_permission("user:delete")(TokenData) → TokenData
4. delete_user(user_id, TokenData) → Response
```

**FastAPI依赖注入的源码级执行流程**：

```python
# fastapi/routing.py 中的调用链（简化版）
class APIRoute:
    async def run(self, request: Request) -> Response:
        # 1. 解析路径参数
        path_params = self.path_converter.convert(request.path_params)

        # 2. 解析依赖（递归执行所有依赖）
        solved_result = await solve_dependencies(
            request=request,
            dependant=self.dependant,
            body_fields=...,
        )

        # 3. 如果任何依赖抛出异常，直接返回错误响应
        if solved_result.errors:
            return self.error_response(solved_result.errors)

        # 4. 用依赖结果调用路由函数
        response = await self.endpoint(**solved_result.values)
        return response
```

**依赖缓存机制**：

```python
# FastAPI在同一请求中缓存依赖结果
# 这意味着 get_current_user 只执行一次，即使多个依赖都需要它

@app.delete("/api/articles/{article_id}")
async def delete_article(
    article_id: str,
    user: TokenData = Depends(get_current_user),       # 执行一次
    checker: TokenData = Depends(require_permission("article:delete")),  # 复用上面的结果
):
    # user 和 checker 是同一个 TokenData 对象
    assert user is checker  # True（同一个请求内）
```

源码中的缓存逻辑：

```python
# fastapi/dependencies/utils.py
async def solve_dependencies(...):
    # 每个请求创建一个新的缓存字典
    cache: Dict[Any, Any] = {}

    for dep in dependant.dependencies:
        # 检查缓存
        cache_key = (dep.call, tuple(sorted(dep.cache_key)))
        if cache_key in cache:
            values[dep.name] = cache[cache_key]
            continue

        # 执行依赖
        result = await dep.call(**sub_values)
        cache[cache_key] = result
        values[dep.name] = result
```

### 15.2.2 权限检查的执行时序

当一个请求到达需要权限验证的路由时，FastAPI按以下时序执行：

```
┌──────────────────────────────────────────────────────────────────┐
│                     HTTP请求到达                                  │
│                   DELETE /api/users/123                          │
│                   Authorization: Bearer <token>                  │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────┐
│ 1. 路由匹配                                                      │
│    FastAPI根据URL和方法匹配到 delete_user 路由                     │
│    确定路径参数 user_id="123"                                      │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────┐
│ 2. 依赖解析（深度优先）                                            │
│    解析路由函数的所有依赖，构建依赖图                                 │
│    确定执行顺序：oauth2_scheme -> get_current_user -> perm_check   │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────┐
│ 3. OAuth2提取（~0.1ms）                                          │
│    从Authorization头提取Bearer Token                              │
│    失败 → 立即返回 401                                            │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────┐
│ 4. JWT验证（~1ms）                                               │
│    jwt.decode() 验证签名和过期时间                                  │
│    查询数据库获取用户信息                                           │
│    失败 → 返回 401                                                │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────┐
│ 5. 权限检查（~5ms）                                               │
│    获取用户角色 → 获取角色权限 → 检查是否包含所需权限                   │
│    失败 → 返回 403                                                │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────┐
│ 6. 动态权限检查（~10ms）                                          │
│    查询资源信息 → 检查资源所有者 → 判定是否有权操作                    │
│    失败 → 返回 403                                                │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────┐
│ 7. 路由函数执行                                                   │
│    所有检查通过，执行业务逻辑                                       │
└──────────────────────────────────────────────────────────────────┘
```

**短路机制**：步骤3-6中任何一步失败，后续步骤全部跳过，直接返回错误响应。这是依赖注入链的核心安全特性——不可能绕过中间环节到达路由函数。

### 15.2.3 与JWT声明的集成

JWT的Payload可以携带角色和权限声明，实现无数据库查询的快速权限检查。这种方案特别适合高并发场景——每次权限检查不需要查询数据库。

**JWT中嵌入角色声明**：

```json
{
  "sub": "user123",
  "roles": ["admin", "editor"],
  "permissions": ["user:create", "user:read", "article:create", "article:publish"],
  "exp": 1701234567
}
```

**Token内的权限 vs 数据库权限的权衡**：

```
Token内权限方案：
+ 优点：无数据库查询，验证速度极快（~1ms）
+ 优点：分布式验证无需共享数据库
- 缺点：权限变更不能即时生效（需等Token过期）
- 缺点：Token体积增大（权限多时可达数KB）

数据库查询方案：
+ 优点：权限变更即时生效
+ 优点：Token体积小
- 缺点：每次请求需查询数据库（~5-10ms）
- 缺点：数据库是性能瓶颈

混合方案（推荐）：
- Token中携带角色（体积小）
- 通过角色查询权限缓存（Redis，~1ms）
- 权限缓存设置短TTL（30-60秒）
- 变更权限时主动刷新缓存
```

**混合方案实现**：

```python
class HybridPermissionChecker:
    """
    混合权限检查器
    - JWT携带角色信息
    - 权限通过角色从缓存/数据库获取
    - 支持即时刷新
    """

    def __init__(self, permission: str):
        self._permission = permission

    async def __call__(
        self,
        current_user: TokenData = Depends(get_current_user),
    ) -> TokenData:
        # 从JWT获取角色（无需数据库查询）
        roles = current_user.roles

        # 从缓存获取角色对应的权限
        permissions = await self._get_role_permissions(roles)

        # 检查权限
        if self._permission not in permissions:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"权限不足：需要 '{self._permission}'",
            )
        return current_user

    async def _get_role_permissions(self, roles: list[str]) -> set[str]:
        """获取角色对应的权限集合（带缓存）"""
        all_permissions: set[str] = set()

        for role in roles:
            # 1. 尝试从Redis缓存获取
            cached = await redis_client.get(f"role_perms:{role}")
            if cached:
                all_permissions.update(json.loads(cached))
                continue

            # 2. 缓存未命中，查询数据库
            perms = await db.query(
                "SELECT p.name FROM permissions p "
                "JOIN role_permissions rp ON p.id = rp.permission_id "
                "JOIN roles r ON rp.role_id = r.id "
                "WHERE r.name = :role",
                {"role": role},
            )
            perm_names = {p["name"] for p in perms}
            all_permissions.update(perm_names)

            # 3. 写入缓存（TTL 60秒）
            await redis_client.setex(
                f"role_perms:{role}",
                60,
                json.dumps(list(perm_names)),
            )

        return all_permissions
```

## 15.3 使用场景

### 15.3.1 多角色系统

典型的多角色系统包含多种用户类型，每种类型有不同的权限集合。角色之间可能存在继承关系。

```
角色继承层次：

    superadmin
        │
        ├── admin
        │     ├── editor
        │     │     ├── author
        │     │     └── moderator
        │     └── analyst
        └── support
              └── viewer

继承规则：
- admin自动拥有editor和analyst的所有权限
- editor自动拥有author和moderator的所有权限
- superadmin拥有所有权限
```

**角色继承的数据模型**：

```sql
-- 角色表
CREATE TABLE roles (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(50) UNIQUE NOT NULL,
    description TEXT,
    parent_id   INTEGER REFERENCES roles(id),  -- 父角色（实现继承）
    created_at  TIMESTAMP DEFAULT NOW()
);

-- 权限表
CREATE TABLE permissions (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(100) UNIQUE NOT NULL,  -- 如 "article:publish"
    description TEXT,
    resource    VARCHAR(50) NOT NULL,           -- 如 "article"
    action      VARCHAR(50) NOT NULL,           -- 如 "publish"
    created_at  TIMESTAMP DEFAULT NOW()
);

-- 角色-权限关联表
CREATE TABLE role_permissions (
    role_id       INTEGER REFERENCES roles(id),
    permission_id INTEGER REFERENCES permissions(id),
    PRIMARY KEY (role_id, permission_id)
);

-- 用户-角色关联表
CREATE TABLE user_roles (
    user_id INTEGER REFERENCES users(id),
    role_id INTEGER REFERENCES roles(id),
    PRIMARY KEY (user_id, role_id)
);
```

### 15.3.2 资源级权限

资源级权限（Resource-Level Permission）是比功能级权限更细粒度的控制。功能级权限回答"用户能否执行此操作"，资源级权限回答"用户能否对此资源执行此操作"。

```
功能级权限：alice 可以删除文章吗？ → 检查 article:delete 权限
资源级权限：alice 可以删除文章#42吗？ → 检查 alice 是否是文章#42的作者 OR alice 有 admin 权限

资源级权限的判定矩阵：
                    文章作者    同组织成员    其他用户
  文章:编辑         ✓           ✓            ✗
  文章:删除         ✓           ✗            ✗
  文章:查看         ✓           ✓            ✓（公开文章）
  文章:查看         ✓           ✓            ✗（私有文章）
```

### 15.3.3 API密钥+角色双认证

某些场景需要双重认证——API密钥标识客户端应用，角色标识用户权限。例如，SaaS平台的API需要同时验证调用方（API密钥）和操作者（用户Token）。

```
请求格式：
GET /api/v1/articles
X-API-Key: ak_live_xxxx          ← 标识客户端应用
Authorization: Bearer eyJhbG...   ← 标识用户身份

验证流程：
1. 验证API密钥 → 确定客户端应用和其配额
2. 验证JWT → 确定用户身份和角色
3. 检查用户角色是否有权执行操作
4. 检查API密钥的配额是否充足
```

```python
# rbac/repository.py - RBAC数据仓库
from typing import Optional, List, Set
from datetime import datetime
import threading


class InMemoryRBACRepository:
    """
    内存版RBAC仓库（用于演示，生产使用SQLAlchemy等ORM）
    实现角色/权限/资源三表模型
    """

    def __init__(self):
        self._roles: dict[int, dict] = {}
        self._permissions: dict[int, dict] = {}
        self._role_permissions: dict[int, set[int]] = {}  # role_id -> {perm_ids}
        self._user_roles: dict[int, set[int]] = {}        # user_id -> {role_ids}
        self._next_role_id = 1
        self._next_perm_id = 1
        self._lock = threading.Lock()

    # --- 角色管理 ---

    def create_role(self, name: str, description: str = "",
                    parent_id: Optional[int] = None) -> dict:
        with self._lock:
            # 检查角色名唯一性
            for role in self._roles.values():
                if role["name"] == name:
                    raise ValueError(f"角色 '{name}' 已存在")

            # 检查父角色存在性
            if parent_id and parent_id not in self._roles:
                raise ValueError(f"父角色ID {parent_id} 不存在")

            role_id = self._next_role_id
            self._next_role_id += 1
            role = {
                "id": role_id,
                "name": name,
                "description": description,
                "parent_id": parent_id,
                "created_at": datetime.utcnow(),
            }
            self._roles[role_id] = role
            self._role_permissions[role_id] = set()
            return role

    def get_role(self, role_id: int) -> Optional[dict]:
        return self._roles.get(role_id)

    def get_role_by_name(self, name: str) -> Optional[dict]:
        for role in self._roles.values():
            if role["name"] == name:
                return role
        return None

    def get_all_roles(self) -> List[dict]:
        return list(self._roles.values())

    # --- 权限管理 ---

    def create_permission(self, name: str, resource: str,
                          action: str, description: str = "") -> dict:
        with self._lock:
            # 检查权限名唯一性
            for perm in self._permissions.values():
                if perm["name"] == name:
                    raise ValueError(f"权限 '{name}' 已存在")

            perm_id = self._next_perm_id
            self._next_perm_id += 1
            perm = {
                "id": perm_id,
                "name": name,
                "resource": resource,
                "action": action,
                "description": description,
            }
            self._permissions[perm_id] = perm
            return perm

    def get_permission(self, perm_id: int) -> Optional[dict]:
        return self._permissions.get(perm_id)

    def get_all_permissions(self) -> List[dict]:
        return list(self._permissions.values())

    # --- 角色-权限关联 ---

    def assign_permission_to_role(self, role_id: int, perm_id: int) -> None:
        with self._lock:
            if role_id not in self._roles:
                raise ValueError(f"角色ID {role_id} 不存在")
            if perm_id not in self._permissions:
                raise ValueError(f"权限ID {perm_id} 不存在")
            self._role_permissions[role_id].add(perm_id)

    def remove_permission_from_role(self, role_id: int, perm_id: int) -> None:
        with self._lock:
            self._role_permissions[role_id].discard(perm_id)

    def get_role_permissions(self, role_id: int) -> List[dict]:
        """获取角色的直接权限（不含继承权限）"""
        perm_ids = self._role_permissions.get(role_id, set())
        return [self._permissions[pid] for pid in perm_ids if pid in self._permissions]

    def get_role_all_permissions(self, role_id: int) -> Set[str]:
        """
        获取角色的所有权限（含继承权限）
        递归遍历角色继承链，合并所有权限
        """
        all_perms: Set[str] = set()
        visited: Set[int] = set()  # 防止循环继承

        def _collect(rid: int):
            if rid in visited:
                return  # 防止循环
            visited.add(rid)

            # 收集直接权限
            perm_ids = self._role_permissions.get(rid, set())
            for pid in perm_ids:
                if pid in self._permissions:
                    all_perms.add(self._permissions[pid]["name"])

            # 递归收集父角色权限
            role = self._roles.get(rid)
            if role and role["parent_id"]:
                _collect(role["parent_id"])

        _collect(role_id)
        return all_perms

    # --- 用户-角色关联 ---

    def assign_role_to_user(self, user_id: int, role_id: int) -> None:
        with self._lock:
            if role_id not in self._roles:
                raise ValueError(f"角色ID {role_id} 不存在")
            if user_id not in self._user_roles:
                self._user_roles[user_id] = set()
            self._user_roles[user_id].add(role_id)

    def remove_role_from_user(self, user_id: int, role_id: int) -> None:
        with self._lock:
            if user_id in self._user_roles:
                self._user_roles[user_id].discard(role_id)

    def get_user_roles(self, user_id: int) -> List[dict]:
        role_ids = self._user_roles.get(user_id, set())
        return [self._roles[rid] for rid in role_ids if rid in self._roles]

    def get_user_all_permissions(self, user_id: int) -> Set[str]:
        """获取用户的所有权限（通过所有角色，含继承）"""
        all_perms: Set[str] = set()
        role_ids = self._user_roles.get(user_id, set())
        for rid in role_ids:
            all_perms.update(self.get_role_all_permissions(rid))
        return all_perms
```

```python
from fastapi import Depends, HTTPException, status
from functools import wraps
from typing import List, Set

from rbac.repository import rbac_repo


class RequirePermission:
    """
    权限检查依赖类
    支持AND/OR语义、通配符匹配
    """

    def __init__(
        self,
        *permissions: str,
        require_all: bool = True,  # True=AND, False=OR
    ):
        self._permissions = permissions
        self._require_all = require_all

    async def __call__(
        self,
        current_user: dict = Depends(get_current_user_with_perms),
    ) -> dict:
        user_perms = current_user.get("permissions", set())

        if self._require_all:
            # AND语义：需要所有权限
            missing = [p for p in self._permissions if not self._match_permission(p, user_perms)]
            if missing:
                raise HTTPException(
                    status_code=status.HTTP_403_FORBIDDEN,
                    detail=f"缺少权限: {', '.join(missing)}",
                )
        else:
            # OR语义：需要任一权限
            if not any(self._match_permission(p, user_perms) for p in self._permissions):
                raise HTTPException(
                    status_code=status.HTTP_403_FORBIDDEN,
                    detail=f"需要以下任一权限: {', '.join(self._permissions)}",
                )

        return current_user

    @staticmethod
    def _match_permission(required: str, user_perms: Set[str]) -> bool:
        """
        权限匹配，支持通配符
        "article:*" 匹配 "article:create", "article:read" 等
        "*:delete" 匹配 "user:delete", "article:delete" 等
        """
        if required in user_perms:
            return True

        # 通配符匹配
        if ":" in required:
            resource, action = required.split(":", 1)
            if action == "*":
                # article:* → 匹配 article:任意
                return any(
                    p.startswith(f"{resource}:") for p in user_perms
                )
            if resource == "*":
                # *:delete → 匹配 任意:delete
                return any(
                    p.endswith(f":{action}") for p in user_perms
                )

        return False
```

```python
# rbac/resource_permissions.py - 资源级权限
from fastapi import Depends, HTTPException, status, Path
from typing import Callable, Optional


class ResourcePermissionChecker:
    """
    资源级权限检查器
    检查用户是否有权操作特定资源
    """

    def __init__(
        self,
        resource_type: str,
        action: str,
        owner_check: bool = True,
        admin_override: str = "admin",
    ):
        self._resource_type = resource_type
        self._action = action
        self._owner_check = owner_check
        self._admin_override = admin_override

    async def __call__(
        self,
        resource_id: str = Path(...),
        current_user: dict = Depends(get_current_user_with_perms),
    ) -> dict:
        # 1. 管理员权限绕过
        required_perm = f"{self._resource_type}:{self._action}"
        admin_perm = f"{self._resource_type}:{self._action}:others"

        user_perms = current_user.get("permissions", set())

        # 拥有 admin_override 权限或 xxx:action:others 权限，直接通过
        if self._admin_override in current_user.get("roles", []) or admin_perm in user_perms:
            return current_user

        # 2. 检查基础权限
        if required_perm not in user_perms:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"权限不足：需要 '{required_perm}'",
            )

        # 3. 资源所有者检查
        if self._owner_check:
            resource = await self._get_resource(resource_id)
            if resource is None:
                raise HTTPException(status_code=404, detail="资源不存在")

            if resource.get("owner_id") != current_user["user_id"]:
                raise HTTPException(
                    status_code=status.HTTP_403_FORBIDDEN,
                    detail="只能操作自己的资源",
                )

        return current_user

    async def _get_resource(self, resource_id: str) -> Optional[dict]:
        """获取资源信息（应从数据库查询）"""
        # 示例：返回模拟数据
        return {"id": resource_id, "owner_id": "1", "title": "示例资源"}
```

```python
# rbac/dynamic_permissions.py - 动态权限系统
from fastapi import Depends, HTTPException, status, Request
from typing import Callable, Any, Optional
from dataclasses import dataclass


@dataclass
class PermissionContext:
    """权限检查上下文 - 传递给权限策略"""
    user_id: str
    roles: list[str]
    permissions: set[str]
    resource_type: str
    resource_id: str
    action: str
    request: Request
    extra: dict  # 额外上下文


class PermissionPolicy:
    """
    权限策略基类
    所有自定义权限策略继承此类
    """

    async def check(self, context: PermissionContext) -> bool:
        """检查权限，返回True表示允许"""
        raise NotImplementedError

    async def get_denial_message(self, context: PermissionContext) -> str:
        """获取拒绝消息"""
        return "权限不足"


class OwnerOrAdminPolicy(PermissionPolicy):
    """资源所有者或管理员策略"""

    async def check(self, context: PermissionContext) -> bool:
        # 管理员直接通过
        if "admin" in context.roles:
            return True

        # 检查是否是资源所有者
        resource = await self._fetch_resource(context)
        if resource and resource.get("owner_id") == context.user_id:
            return True

        return False

    async def _fetch_resource(self, context: PermissionContext) -> Optional[dict]:
        # 实际应从数据库查询
        return {"owner_id": context.user_id}


class OrganizationMemberPolicy(PermissionPolicy):
    """组织成员策略 - 同组织成员可操作"""

    async def check(self, context: PermissionContext) -> bool:
        if "admin" in context.roles:
            return True

        # 检查用户和资源是否属于同一组织
        user_orgs = await self._get_user_orgs(context.user_id)
        resource_org = await self._get_resource_org(context.resource_id)

        if resource_org in user_orgs:
            return True

        return False

    async def _get_user_orgs(self, user_id: str) -> set[str]:
        return {"org-1"}  # 模拟

    async def _get_resource_org(self, resource_id: str) -> str:
        return "org-1"  # 模拟


class RateLimitPolicy(PermissionPolicy):
    """速率限制策略 - 基于用户角色的不同速率"""

    async def check(self, context: PermissionContext) -> bool:
        # 获取用户角色的速率限制
        limits = {
            "admin": 1000,
            "editor": 100,
            "viewer": 10,
        }

        user_limit = 0
        for role in context.roles:
            user_limit = max(user_limit, limits.get(role, 0))

        # 检查当前使用量（应从Redis查询）
        current_usage = await self._get_usage(context.user_id, context.action)
        return current_usage < user_limit

    async def _get_usage(self, user_id: str, action: str) -> int:
        return 5  # 模拟


class DynamicPermissionChecker:
    """
    动态权限检查器
    组合多个策略，按顺序检查
    """

    def __init__(self, resource_type: str, action: str, *policies: PermissionPolicy):
        self._resource_type = resource_type
        self._action = action
        self._policies = policies

    async def __call__(
        self,
        request: Request,
        resource_id: str = Path(...),
        current_user: dict = Depends(get_current_user_with_perms),
    ) -> dict:
        context = PermissionContext(
            user_id=current_user["user_id"],
            roles=current_user["roles"],
            permissions=current_user["permissions"],
            resource_type=self._resource_type,
            resource_id=resource_id,
            action=self._action,
            request=request,
            extra={},
        )

        for policy in self._policies:
            allowed = await policy.check(context)
            if not allowed:
                message = await policy.get_denial_message(context)
                raise HTTPException(
                    status_code=status.HTTP_403_FORBIDDEN,
                    detail=message,
                )

        return current_user
```

**陷阱：权限粒度过细**

```python
# 错误：每个字段一个权限
permissions = [
    "user:update:username",
    "user:update:email",
    "user:update:password",
    "user:update:avatar",
    "user:update:bio",
    # ... 如果用户有20个字段，就需要20个权限
]

# 正确：字段分组
permissions = [
    "user:update:profile",   # 用户名、头像、简介
    "user:update:email",     # 邮箱（敏感，单独控制）
    "user:update:password",  # 密码（最敏感，单独控制）
]
```

**最佳实践：三层粒度模型**

```
Level 1: 资源级（粗粒度） - 用于路由级访问控制
  article:read  → 可查看任何文章

Level 2: 操作级（中粒度） - 用于功能级权限控制
  article:update → 可编辑文章
  article:update:others → 可编辑他人文章

Level 3: 字段级（细粒度） - 用于数据级权限控制
  article:update:status → 可修改文章状态（发布/下架）
  article:update:content → 可修改文章内容

原则：从粗到细，根据实际需求选择粒度
不要过度设计 - 大多数场景Level 1 + Level 2足够
```

### 15.5.2 权限缓存

**陷阱：权限变更不即时生效**

```python

# 如果管理员撤销了某用户的权限，用户在1小时内仍然可以操作！
```

**最佳实践：主动失效 + 短TTL**

```python
class PermissionCache:
    """权限缓存 - 支持主动失效"""

    def __init__(self, redis_client, default_ttl: int = 60):
        self._redis = redis_client
        self._ttl = default_ttl

    async def get_role_permissions(self, role: str) -> set[str] | None:
        """获取角色权限（从缓存）"""
        cached = await self._redis.get(f"role_perms:{role}")
        if cached:
            return set(json.loads(cached))
        return None

    async def set_role_permissions(self, role: str, permissions: set[str]) -> None:
        """设置角色权限缓存"""
        await self._redis.setex(
            f"role_perms:{role}",
            self._ttl,
            json.dumps(list(permissions)),
        )

    async def invalidate_role(self, role: str) -> None:
        """主动失效某角色的缓存"""
        await self._redis.delete(f"role_perms:{role}")

    async def invalidate_user(self, user_id: str) -> None:
        """主动失效某用户的所有权限缓存"""
        # 删除用户权限缓存
        await self._redis.delete(f"user_perms:{user_id}")

    async def invalidate_all(self) -> None:
        """失效所有权限缓存（批量操作）"""
        # 使用SCAN而非KEYS（避免阻塞Redis）
        cursor = 0
        while True:
            cursor, keys = await self._redis.scan(
                cursor, match="role_perms:*", count=100
            )
            if keys:
                await self._redis.delete(*keys)
            if cursor == 0:
                break
```

### 15.5.3 权限变更的实时生效

**陷阱：JWT中的角色信息在Token签发后不可变**

```python
# 错误：仅依赖JWT中的角色信息
# 如果管理员撤销了用户的admin角色，但用户的JWT仍包含roles=["admin"]
# 用户在Token过期前仍然拥有admin权限！
```

**最佳实践：版本号机制实现实时权限变更**

```python
class UserPermissionVersion:
    """
    用户权限版本号
    每次用户权限变更时递增版本号
    JWT中携带签发时的版本号，验证时与当前版本号比较
    """

    def __init__(self, redis_client):
        self._redis = redis_client
        self._prefix = "user_perm_version:"

    async def get_version(self, user_id: str) -> int:
        """获取用户当前的权限版本号"""
        val = await self._redis.get(f"{self._prefix}{user_id}")
        return int(val) if val else 0

    async def increment_version(self, user_id: str) -> int:
        """递增用户权限版本号（权限变更时调用）"""
        return await self._redis.incr(f"{self._prefix}{user_id}")


# JWT签发时携带版本号
async def create_token(user_id: str, roles: list[str]) -> str:
    version = await perm_version.get_version(user_id)
    payload = {
        "sub": user_id,
        "roles": roles,
        "perm_v": version,  # 权限版本号
        "exp": datetime.utcnow() + timedelta(minutes=15),
    }
    return jwt.encode(payload, SECRET_KEY, algorithm="HS256")


# Token验证时检查版本号
async def get_current_user(token: str = Depends(oauth2_scheme)) -> dict:
    payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])

    # 检查权限版本号
    current_version = await perm_version.get_version(payload["sub"])
    if payload.get("perm_v", 0) < current_version:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="权限已变更，请重新登录",
        )

    return payload
```

**最佳实践：权限变更通知机制**

```python
class PermissionChangeNotifier:
    """
    权限变更通知器
    通过WebSocket或SSE实时通知前端权限变更
    """

    def __init__(self):
        self._connections: dict[str, list[WebSocket]] = {}

    async def subscribe(self, user_id: str, websocket: WebSocket):
        """用户订阅权限变更通知"""
        await websocket.accept()
        if user_id not in self._connections:
            self._connections[user_id] = []
        self._connections[user_id].append(websocket)

    async def notify_user(self, user_id: str, change: dict):
        """通知用户权限变更"""
        connections = self._connections.get(user_id, [])
        dead = []
        for ws in connections:
            try:
                await ws.send_json({
                    "type": "permission_changed",
                    "data": change,
                })
            except Exception:
                dead.append(ws)
        for ws in dead:
            connections.remove(ws)

    async def notify_role_change(self, role_name: str, change_type: str):
        """通知所有拥有该角色的用户"""
        # 查询拥有该角色的所有用户
        user_ids = await db.query(
            "SELECT user_id FROM user_roles ur "
            "JOIN roles r ON ur.role_id = r.id "
            "WHERE r.name = :role",
            {"role": role_name},
        )
        for uid in user_ids:
            await self.notify_user(str(uid["user_id"]), {
                "type": change_type,
                "role": role_name,
                "message": "您的权限已变更，请刷新页面",
            })
```

**最佳实践总结**：

1. **权限粒度**：三层粒度模型（资源级/操作级/字段级），按需选择，避免过粗或过细
2. **权限缓存**：短TTL（30-60秒）+ 主动失效，变更时立即刷新缓存
3. **实时生效**：版本号机制 + Token中嵌入版本号，变更时递增使旧Token失效
4. **角色继承**：支持角色继承链，但防止循环继承
5. **通配符**：支持`article:*`通配符匹配，简化常见权限组合
6. **资源级权限**：对于"只能操作自己的资源"场景，使用动态权限检查
7. **双认证**：API密钥+角色双认证用于SaaS场景
8. **权限审计**：记录所有权限变更的审计日志
