### 30.1.2 事务回滚模式

```python
"""
事务回滚测试模式（推荐）：

1. 每个测试在事务中运行
2. 测试结束后回滚事务
3. 数据库始终保持干净状态
4. 无需创建/删除表

优点：
- 测试隔离性好
- 测试速度快（无需重建表）
- 数据库状态可预测
"""
import pytest
from sqlalchemy.ext.asyncio import AsyncSession

@pytest.fixture
async def db_session(engine):
    """每个测试使用独立事务。"""
    connection = await engine.connect()
    transaction = await connection.begin()
    async_session = sessionmaker(connection, class_=AsyncSession)

    async with async_session() as session:
        yield session

    # 回滚事务，恢复干净状态
    await transaction.rollback()
    await connection.close()

@pytest.fixture
def client(db_session):
    """测试客户端使用事务数据库。"""
    async def override_get_db():
        yield db_session

    app.dependency_overrides[get_db] = override_get_db
    with TestClient(app) as c:
        yield c
    app.dependency_overrides.clear()
```

### 30.1.3 测试数据工厂

```python
from factory import Factory, LazyFunction, Sequence
import random

class UserFactory(Factory):
    class Meta:
        model = User

    id = Sequence(lambda n: n + 1)
    name = Sequence(lambda n: f"user_{n}")
    email = Sequence(lambda n: f"user_{n}@example.com")
    role = "user"
    is_active = True

class ItemFactory(Factory):
    class Meta:
        model = Item

    id = Sequence(lambda n: n + 1)
    name = Sequence(lambda n: f"Item {n}")
    price = LazyFunction(lambda: round(random.uniform(1, 100), 2))
    stock = LazyFunction(lambda: random.randint(0, 100))

# 使用工厂创建测试数据
def test_list_items(client, db_session):
    # 创建测试数据
    for i in range(5):
        item = ItemFactory.create()
        db_session.add(item)
    await db_session.commit()

    response = client.get("/items")
    assert response.status_code == 200
    assert len(response.json()) == 5
```

## 30.2 原理与机制

### 30.2.1 测试数据库的隔离策略

```python
"""
测试数据库隔离策略对比：

1. 内存 SQLite（推荐简单项目）：
   - 速度最快
   - 无需外部依赖
   - 不支持 PostgreSQL 特有功能

2. Docker PostgreSQL（推荐生产项目）：
   - 与生产环境一致
   - 支持 PostgreSQL 特有功能
   - 需要 Docker 环境

3. 事务回滚（推荐配合上述两者）：
   - 每个测试在事务中运行
   - 测试后回滚
   - 保持数据库干净

4. 测试数据库配置最佳实践：
   - 使用独立数据库（非生产）
   - 每次 CI 运行创建新数据库
   - 使用 Alembic 迁移确保结构一致
   - 测试后清理数据

Docker 测试数据库配置：
```yaml
# docker-compose.test.yml
services:
  test-db:
    image: postgres:15
    environment:
      POSTGRES_DB: test_db
      POSTGRES_USER: test
      POSTGRES_PASSWORD: test
    ports:
      - "5433:5432"
```
"""
```

### 30.2.2 集成测试的边界

```python
"""
集成测试 vs 单元测试：

单元测试：
- 测试单个函数/类
- 模拟所有外部依赖
- 速度快，隔离性好
- 不验证组件间协作

集成测试：
- 测试多个组件协作
- 使用真实数据库（或测试数据库）
- 可能涉及外部服务（模拟或真实）
- 验证端到端流程

集成测试应该覆盖：
1. API 端到端：HTTP 请求 → 数据库 → 响应
2. 认证流程：注册 → 登录 → 访问受保护资源
3. 数据完整性：创建 → 查询 → 更新 → 删除
4. 并发场景：多个请求同时操作同一数据
5. 错误传播：数据库错误 → API 错误响应

不应该在集成测试中测试：
- 业务逻辑的边界条件（应单独单元测试）
- Pydantic 验证（应单独测试）
- 外部 API 的调用（应模拟）
"""
```

## 30.3 使用场景

### 30.3.1 CRUD 集成测试

```python
import pytest
from fastapi.testclient import TestClient

class TestUserCRUD:
    """用户 CRUD 集成测试。"""

    def test_create_user(self, client, db_session):
        response = client.post("/users", json={
            "name": "Alice", "email": "alice@example.com", "age": 30,
        })
        assert response.status_code == 201
        data = response.json()
        assert data["name"] == "Alice"
        assert "id" in data

    def test_get_user(self, client, db_session):
        # 先创建
        create_resp = client.post("/users", json={
            "name": "Bob", "email": "bob@example.com", "age": 25,
        })
        user_id = create_resp.json()["id"]

        # 再查询
        response = client.get(f"/users/{user_id}")
        assert response.status_code == 200
        assert response.json()["name"] == "Bob"

    def test_update_user(self, client, db_session):
        create_resp = client.post("/users", json={
            "name": "Charlie", "email": "charlie@example.com", "age": 35,
        })
        user_id = create_resp.json()["id"]

        response = client.patch(f"/users/{user_id}", json={"name": "Charles"})
        assert response.status_code == 200
        assert response.json()["name"] == "Charles"

    def test_delete_user(self, client, db_session):
        create_resp = client.post("/users", json={
            "name": "Dave", "email": "dave@example.com", "age": 40,
        })
        user_id = create_resp.json()["id"]

        response = client.delete(f"/users/{user_id}")
        assert response.status_code == 204

        # 确认已删除
        response = client.get(f"/users/{user_id}")
        assert response.status_code == 404
```

### 30.3.2 认证流程集成测试

class TestAuthFlow:
    """认证流程集成测试。"""

    def test_register_login_access(self, client, db_session):
        # 1. 注册
        register_resp = client.post("/auth/register", json={
            "username": "testuser", "email": "test@example.com",
            "password": "Test@1234",
        })
        assert register_resp.status_code == 201

        # 2. 登录
        login_resp = client.post("/auth/login", json={
            "username": "testuser", "password": "Test@1234",
        })
        assert login_resp.status_code == 200
        token = login_resp.json()["access_token"]

        # 3. 访问受保护资源
        headers = {"Authorization": f"Bearer {token}"}
        me_resp = client.get("/me", headers=headers)
        assert me_resp.status_code == 200
        assert me_resp.json()["username"] == "testuser"

        # 4. 无效 token 访问
        bad_headers = {"Authorization": "Bearer invalid_token"}
        bad_resp = client.get("/me", headers=bad_headers)
        assert bad_resp.status_code == 401

## 30.5 常见陷阱与最佳实践

| 陷阱 | 最佳实践 |
|------|---------|
| 测试使用生产数据库 | 使用独立的测试数据库 |
| 测试间数据相互影响 | 使用事务回滚确保隔离 |
| 不清理依赖覆盖 | 使用 fixture yield 确保清理 |
| 不测试数据库迁移 | 在 CI 中运行 Alembic 迁移测试 |
| 模拟过多依赖 | 集成测试尽量使用真实组件 |
| 不测试并发场景 | 使用 asyncio.gather 模拟并发请求 |
| 测试顺序依赖 | 每个测试独立，不依赖执行顺序 |
