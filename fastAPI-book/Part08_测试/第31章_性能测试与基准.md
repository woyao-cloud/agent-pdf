### 31.1.3 性能指标

```python
"""
关键性能指标：

1. 延迟（Latency）：
   - P50：50% 请求的响应时间
   - P90：90% 请求的响应时间
   - P99：99% 请求的响应时间
   - 目标：P99 < 500ms（API 类应用）

2. 吞吐量（Throughput）：
   - RPS：每秒请求数
   - 目标：根据业务需求设定

3. 错误率：
   - 非正常状态码的请求比例
   - 目标：< 0.1%

4. 并发连接数：
   - 同时活跃的连接数
   - 目标：根据 Worker 数量设定

5. 资源使用：
   - CPU 使用率
   - 内存使用量
   - 数据库连接池使用率
"""
```

## 31.2 原理与机制

### 31.2.1 FastAPI 性能瓶颈分析

```python
"""
FastAPI 请求处理中的性能瓶颈：

1. 同步 I/O 阻塞（最大瓶颈）：
   - 同步数据库查询（如 psycopg2）
   - 同步 HTTP 请求（如 requests）
   - 同步文件 I/O
   - 解决：使用异步库或 run_in_threadpool

2. 数据库查询：
   - N+1 查询问题
   - 缺少索引
   - 大量数据未分页
   - 解决：使用 eager loading、索引、分页

3. 序列化开销：
   - 大量模型序列化
   - response_model 过滤
   - 解决：优化模型、使用 exclude

4. 中间件栈：
   - 过多的中间件
   - 中间件中的阻塞操作
   - 解决：减少中间件、使用 ASGI 中间件

5. 依赖注入：
   - 依赖链过长
   - 依赖中执行耗时操作
   - 解决：缓存依赖、精简依赖链

6. Pydantic 验证：
   - 复杂嵌套模型
   - 大量自定义验证器
   - 解决：使用 strict 模式、简化模型

性能分析工具：
- py-spy：Python 进程采样分析器
- cProfile：标准库性能分析
- Austin：帧栈采样分析器
- New Relic / Datadog：APM 监控
"""
```

### 31.2.2 事件循环与性能

```python
"""
asyncio 事件循环对性能的影响：

1. 事件循环是单线程的：
   - 任何阻塞操作都会影响所有请求
   - 同步 I/O 是最大敌人
   - CPU 密集型任务也会阻塞

2. 协程切换开销：
   - 每个协程切换约 1-2 微秒
   - 大量微协程可能累积开销
   - asyncio.gather 的调度开销

3. 最佳 Worker 配置：
   - CPU 密集型：workers = CPU 核心数
   - I/O 密集型：workers = CPU 核心数 * 2 + 1
   - 使用 uvloop 提升事件循环性能

4. 关键性能优化：
   - 使用 uvloop（替代 asyncio 默认事件循环）
   - 使用 httptools（替代 h11 HTTP 解析器）
   - 使用 orjson（替代 json 序列化）
   - 使用 asyncio 的 run_in_threadpool 处理同步操作
   - 合理使用连接池（数据库、HTTP）

性能基准参考（单 Worker，简单 JSON 响应）：
- FastAPI + uvicorn: ~30,000 RPS
- FastAPI + uvicorn + uvloop: ~40,000 RPS
- FastAPI + uvicorn + uvloop + httptools: ~50,000 RPS
"""
```

## 31.3 使用场景

### 31.3.1 API 性能基准测试

```python
"""使用 pytest-benchmark 的性能测试套件"""
import pytest
from fastapi.testclient import TestClient

    def test_get_item_performance(self, benchmark):
        result = benchmark(client.get, "/items/1")
        assert result.status_code == 200

    @pytest.mark.benchmark(group="write", min_rounds=50)
    def test_create_item_performance(self, benchmark):
        def create():
            return client.post("/items", json={
                "name": "Perf Item", "price": 9.99,
            })
        result = benchmark(create)
        assert result.status_code == 201

    @pytest.mark.benchmark(group="auth", min_rounds=100)
    def test_auth_performance(self, benchmark):

            return client.post("/auth/login", json={
                "username": "test", "password": "test",
            })
        result = benchmark(auth)
        assert result.status_code == 200
```

### 31.3.2 并发测试

```python
"""使用 asyncio 进行并发测试"""
import asyncio
import pytest
import httpx

@pytest.mark.asyncio
async def test_concurrent_requests():
    """测试并发请求处理。"""
    async with httpx.AsyncClient(base_url="http://localhost:8000") as client:
        tasks = [client.get("/items") for _ in range(100)]
        responses = await asyncio.gather(*tasks)

        # 所有请求应成功
        for response in responses:
            assert response.status_code == 200

@pytest.mark.asyncio
async def test_concurrent_writes():
    """测试并发写入（检查数据一致性）。"""
    async with httpx.AsyncClient(base_url="http://localhost:8000") as client:
        tasks = [
            client.post("/items", json={"name": f"Item {i}", "price": i * 1.0})
            for i in range(50)
        ]
        responses = await asyncio.gather(*tasks)

        # 统计成功和失败
        successes = sum(1 for r in responses if r.status_code == 201)
        assert successes == 50
```

## 31.5 常见陷阱与最佳实践

| 陷阱 | 最佳实践 |
|------|---------|
| 在开发环境测试性能 | 使用生产配置（uvloop、多 Worker）进行性能测试 |
| 只测平均延迟 | 关注 P90/P99 延迟 |
| 不预热服务 | 性能测试前先发送预热请求 |
| 不关闭调试模式 | 生产测试关闭 debug 和 reload |
| 忽略数据库性能 | 数据库查询往往是最大瓶颈 |
| 不测试并发场景 | 模拟真实并发负载 |
| 连接池配置不当 | 根据负载调整数据库连接池大小 |
| 不做持续性能监控 | 集成到 CI/CD 流水线 |
