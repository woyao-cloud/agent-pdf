# 第34章 监控与可观测性

## 34.1 概念与语法

### 34.1.1 可观测性三支柱

可观测性（Observability）是系统理解自身内部状态的能力，建立在三大支柱之上：

| 支柱 | 关注点 | 典型工具 | FastAPI 中的体现 |
|------|--------|----------|-----------------|
| Metrics（指标） | 系统的量化状态 | Prometheus, Grafana | 请求延迟、错误率、吞吐量 |
| Logs（日志） | 离散事件的记录 | ELK, Loki | 请求日志、错误日志、业务日志 |
| Traces（追踪） | 请求在系统中的流转路径 | Jaeger, Zipkin | 跨服务的请求链路追踪 |

三者的关系：

```
Metrics: "有多少请求失败了？" → 告警触发
  │
  ├── Logs: "具体的错误是什么？" → 定位错误详情
  │     │
  │     └── Traces: "错误发生在哪个服务的哪个调用？" → 定位根因
```

### 34.1.2 Prometheus 指标类型

Prometheus 定义四种核心指标类型：

**Counter（计数器）**：单调递增，只增不减。用于累计值（请求总数、错误总数）。

```python
from prometheus_client import Counter

REQUEST_COUNT = Counter(
    "http_requests_total",
    "Total count of HTTP requests",
    ["method", "endpoint", "status_code"]
)

# 使用: 值只能增加
REQUEST_COUNT.labels(method="GET", endpoint="/api/users", status_code=200).inc()
```

**Gauge（仪表盘）**：可增可减。用于当前状态值（当前连接数、队列长度）。

```python
from prometheus_client import Gauge

ACTIVE_CONNECTIONS = Gauge(
    "active_connections",
    "Number of active connections"
)

# 使用: 可增可减
ACTIVE_CONNECTIONS.inc()   # +1
ACTIVE_CONNECTIONS.dec()   # -1
ACTIVE_CONNECTIONS.set(42) # 设为 42
```

**Histogram（直方图）**：对观测值采样并统计分布。用于请求延迟等。

```python
from prometheus_client import Histogram

REQUEST_DURATION = Histogram(
    "http_request_duration_seconds",
    "HTTP request duration in seconds",
    ["method", "endpoint"],
    buckets=[0.01, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0, 10.0]
)

# 使用: 观测单个值
with REQUEST_DURATION.labels(method="GET", endpoint="/api/users").time():
    # 处理请求
    pass
```

Histogram 会自动生成三类时间序列：
- `_bucket`：每个桶的累计计数（用于计算百分位数）
- `_sum`：观测值总和
- `_count`：观测次数

**Summary（摘要）**：在客户端计算百分位数。与 Histogram 不同，Summary 在应用进程内计算分位数，但无法跨实例聚合。

```python
from prometheus_client import Summary

REQUEST_DURATION_SUMMARY = Summary(
    "http_request_duration_summary_seconds",
    "HTTP request duration summary",
    ["method"]
)
```

### 34.1.3 OpenTelemetry 概念体系

OpenTelemetry（OTel）是 CNCF 的可观测性标准，统一了 Metrics、Logs、Traces 三种信号的采集和传输。

核心概念：

| 概念 | 说明 |
|------|------|
| **Tracer** | 创建 Span 的工厂 |
| **Span** | 一次操作的时间段，含名称、开始时间、持续时间、属性 |
| **Context** | 跨进程传递追踪信息的载体（Trace Context） |
| **Propagator** | 在进程边界注入/提取 Context 的机制 |
| **Exporter** | 将遥测数据发送到后端（Jaeger, Prometheus 等） |
| **Resource** | 描述产生遥测数据的实体（服务名、版本等） |
| **Instrumentation** | 自动或手动采集遥测数据的组件 |

Span 的结构：

```
Trace ID: abc123
├── Span: HTTP GET /api/users (root span)
│   ├── Span: DB query - SELECT users
│   ├── Span: Redis GET cache:users
│   └── Span: HTTP POST http://auth-service/verify
│       └── Span: DB query - SELECT tokens
```

W3C Trace Context 传播格式：

```
traceparent: 00-<trace-id>-<span-id>-<trace-flags>
tracestate: vendor-specific-data

示例:
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
```

### 34.1.4 结构化日志格式

结构化日志将日志消息编码为机器可解析的结构化格式（通常为 JSON），而非人类可读的纯文本。

```json
{
  "timestamp": "2024-01-15T10:30:00.123Z",
  "level": "ERROR",
  "logger": "app.api.users",
  "message": "Failed to fetch user profile",
  "request_id": "req-abc123",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id": "00f067aa0ba902b7",
  "user_id": 42,
  "endpoint": "/api/users/42",
  "method": "GET",
  "status_code": 500,
  "duration_ms": 150.3
}
```

Python structlog 语法：

```python
import structlog

logger = structlog.get_logger()

# 结构化日志记录
logger.info("request_completed",
            method="GET",
            endpoint="/api/users",
            status_code=200,
            duration_ms=42.5)

# 绑定上下文
log = logger.bind(request_id="req-abc123", user_id=42)
log.error("database_connection_failed", error_code="ECONNREFUSED")
```

### 34.1.5 健康检查端点设计

Kubernetes 和 Docker 使用两类探针（probe）：

| 探针类型 | 用途 | 失败后果 |
|----------|------|----------|
| **Liveness Probe** | 进程是否存活 | 重启容器 |
| **Readiness Probe** | 是否可处理请求 | 从 Service 端点列表移除 |
| **Startup Probe** | 是否已启动完成 | 在 startup 前，liveness 探针暂停 |

```python
from fastapi import FastAPI
from fastapi.responses import JSONResponse

app = FastAPI()

@app.get("/health/live")
async def liveness():
    """进程存活检查 - 不检查依赖"""
    return {"status": "alive"}

@app.get("/health/ready")
async def readiness():
    """就绪检查 - 检查所有依赖"""
    checks = {}
    try:
        await check_database()
        checks["database"] = "ok"
    except Exception as e:
        checks["database"] = f"error: {e}"

    all_ok = all(v == "ok" for v in checks.values())
    if not all_ok:
        return JSONResponse(status_code=503, content={"status": "not_ready", "checks": checks})
    return {"status": "ready", "checks": checks}
```

## 34.2 原理与机制

### 34.2.1 Prometheus 拉取模型

Prometheus 采用 **Pull 模型**：Prometheus Server 主动拉取（scrape）目标的 `/metrics` 端点，而非应用推送数据。

```
┌─────────────┐    scrape     ┌──────────────┐
│ Prometheus  │ ◄──────────── │ FastAPI App  │
│   Server    │    /metrics   │  :8000       │
└─────────────┘               └──────────────┘
       │
       │ scrape
       ▼
┌──────────────┐
│ FastAPI App  │
│  :8001       │
└──────────────┘
```

**Pull 模型的优势**：

1. 服务发现：Prometheus 可通过 DNS、K8s API 等自动发现新实例
2. 目标健康监控：拉取失败本身就是监控信号
3. 无需在应用中配置推送地址
4. 避免推送洪峰：Prometheus 控制拉取频率

**拉取过程**：

1. Prometheus 根据 `scrape_interval`（默认 15s）定时发起 HTTP GET 请求到目标
2. 目标返回文本格式的指标数据
3. Prometheus 解析文本并存储到时序数据库（TSDB）

**文本格式示例**：

```text
# HELP http_requests_total Total count of HTTP requests
# TYPE http_requests_total counter
http_requests_total{method="GET",endpoint="/api/users",status_code="200"} 1542
http_requests_total{method="POST",endpoint="/api/users",status_code="201"} 123
http_requests_total{method="GET",endpoint="/api/users",status_code="500"} 3

# TYPE http_request_duration_seconds histogram
http_request_duration_seconds_bucket{method="GET",endpoint="/api/users",le="0.01"} 100
http_request_duration_seconds_bucket{method="GET",endpoint="/api/users",le="0.05"} 450
http_request_duration_seconds_bucket{method="GET",endpoint="/api/users",le="0.1"} 800
http_request_duration_seconds_bucket{method="GET",endpoint="/api/users",le="+Inf"} 1542
http_request_duration_seconds_sum{method="GET",endpoint="/api/users"} 89.3
http_request_duration_seconds_count{method="GET",endpoint="/api/users"} 1542
```

### 34.2.2 OpenTelemetry SDK 架构

OpenTelemetry Python SDK 的内部架构分为三层：

```
应用层
  │
  ├── OTel API (opentelemetry-api)
  │     Tracer, Meter, Logger  ← 开发者使用的接口
  │
  ├── OTel SDK (opentelemetry-sdk)
  │     ├── TracerProvider → 创建 Span
  │     ├── MeterProvider → 创建 Instruments
  │     ├── LoggerProvider → 创建 LogRecord
  │     └── Processing Pipeline:
  │           SpanProcessor → BatchSpanProcessor → Exporter
  │
  └── Exporter
        ├── OTLP Exporter → OTel Collector
        ├── Prometheus Exporter → Prometheus Server
        └── Jaeger Exporter → Jaeger Backend
```

**Span 处理管线**：

```
Span 创建
  │
  ├── SpanProcessor.on_start()  ← Span 开始时调用
  │     (添加属性、采样决策)
  │
  ├── Span 结束
  │
  └── SpanProcessor.on_end()    ← Span 结束时调用
        │
        ├── SimpleSpanProcessor: 直接调用 Exporter
        └── BatchSpanProcessor: 缓冲后批量调用 Exporter
              │
              └── Exporter.export() → 发送到后端
```

**BatchSpanProcessor** 是生产环境的推荐选择。它将 Span 缓冲到队列中，定期批量发送，减少网络 I/O 开销。配置参数：

```python
from opentelemetry.sdk.trace.export import BatchSpanProcessor

processor = BatchSpanProcessor(
    exporter=otlp_exporter,
    max_queue_size=2048,        # 队列最大容量
    schedule_delay_millis=5000, # 批量发送间隔（毫秒）
    max_export_batch_size=512,  # 单次最大导出数量
    export_timeout_millis=30000 # 导出超时
)
```

### 34.2.3 采样策略

在高流量系统中，采集所有 Span 会产生巨大的数据量和性能开销。OpenTelemetry 支持多种采样策略：

| 采样器 | 策略 | 适用场景 |
|--------|------|----------|
| `AlwaysOn` | 采集所有 | 开发环境 |
| `AlwaysOff` | 不采集 | 测试环境 |
| `TraceIdRatio` | 按比例采样 | 生产环境（如 10%） |
| `ParentBased` | 根据父 Span 决策 | 保持链路完整 |
| `TailBased` | 根据结果决策 | 仅采集错误/慢请求 |

**ParentBased 采样器**的关键逻辑：

```python
# 简化逻辑
class ParentBasedSampler:
    def should_sample(self, parent_context, trace_id, ...):
        if parent_context:
            # 如果父 Span 已采样，子 Span 也采样（保持链路完整）
            if parent_context.trace_flags.sampled:
                return RECORD_AND_SAMPLE
            else:
                return DROP
        else:
            # 根 Span 由 root_sampler 决策
            return self.root_sampler.should_sample(...)
```

### 34.2.4 Context Propagation 机制

分布式追踪的核心挑战是在跨服务调用时传递 Trace Context。OpenTelemetry 使用 **Propagator** 在进程边界注入和提取 Context。

**HTTP 传播**（W3C Trace Context）：

```
服务 A ──HTTP──► 服务 B
  │                 │
  ├── Inject:       ├── Extract:
  │   traceparent   │   traceparent
  │   → Header      │   → Context
  │                 │
  └─────────────────┘
```

```python
# 注入（发送方）
from opentelemetry.propagate import inject

headers = {}
inject(headers)  # 将当前 Context 写入 headers
# headers = {"traceparent": "00-abc123-def456-01"}

# 提取（接收方）
from opentelemetry.propagate import extract

carrier = {"traceparent": "00-abc123-def456-01"}
context = extract(carrier)  # 从 headers 恢复 Context
```

**FastAPI 中间件自动传播**：

```python
# OpenTelemetry FastAPI Instrumentation 自动处理
# 在 ASGI 中间件层自动从请求头提取 Context
# 并在响应头注入 Context

async def asgi_middleware(scope, receive, send):
    if scope["type"] == "http":
        # 从请求头提取 Trace Context
        headers = dict(scope.get("headers", []))
        context = extract(headers)
        # 在当前 Context 下执行应用
        with tracer.start_as_current_span("request", context=context):
            await app(scope, receive, send)
```

### 34.2.5 structlog 的处理管线

structlog 的日志输出经过一条可配置的处理管线：

```
日志事件
  │
  ├── 1. Pre-chain processors (上下文绑定)
  │     - 添加 timestamp
  │     - 添加 log_level
  │     - 添加 request_id
  │     - 添加 trace_id / span_id
  │

  │     - JSONRenderer: 输出 JSON
  │     - ConsoleRenderer: 终端彩色输出

  │
  └── 输出到 stdout/stderr
```

```python
import structlog

structlog.configure(
```

### 34.2.6 Histogram 与百分位数计算

Prometheus 通过 Histogram 的 `_bucket` 数据在服务端计算百分位数，使用 `histogram_quantile` 函数：

```
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
```

计算原理：

1. 每个 bucket 记录小于等于其 le 值的累计请求数
2. `histogram_quantile` 在 bucket 之间线性插值
3. 精度取决于 bucket 边界的密度

```
bucket 分布示例:
le=0.01  : 100  (100 个请求 ≤ 10ms)
le=0.05  : 450  (350 个请求在 10-50ms)
le=0.1   : 800  (350 个请求在 50-100ms)
le=0.25  : 1200 (400 个请求在 100-250ms)
le=0.5   : 1400 (200 个请求在 250-500ms)
le=1.0   : 1500 (100 个请求在 500ms-1s)
le=+Inf  : 1542 (42 个请求 > 1s)

P95 计算: 总数 * 0.95 = 1464.9
  → 落在 le=0.5 和 le=1.0 之间
  → 线性插值: 0.5 + (1464.9 - 1400)/(1500-1400) * (1.0-0.5) = 0.82s
```

## 34.3 使用场景

### 34.3.1 生产环境全链路监控

在微服务架构中，一个 API 请求可能经过多个服务。全链路追踪让开发者能看到请求的完整路径和每个环节的耗时。

### 34.3.2 告警驱动的故障发现

通过 Prometheus 告警规则，在指标异常时自动触发通知：

- 5xx 错误率超过 1% → P0 告警
- P99 延迟超过 2s → P1 告警
- 数据库连接池使用率超过 80% → P2 告警

### 34.3.3 性能回归检测

通过 Histogram 追踪请求延迟分布，检测部署后的性能回归：

```
部署前: P50=50ms, P95=200ms, P99=500ms
部署后: P50=55ms, P95=350ms, P99=2000ms  ← 性能回归!
```

### 34.3.4 容量规划与弹性伸缩

通过 Metrics 历史数据分析资源使用趋势，为容量规划提供数据支撑：

- CPU 使用率趋势
- 内存使用趋势
- 请求 QPS 趋势

### 34.3.5 故障根因分析

通过 Traces + Logs 的关联，快速定位故障根因：

1. 告警触发：API 错误率上升
2. 查看 Metrics：确认哪些端点受影响
3. 查看 Traces：发现受影响请求都卡在数据库查询
4. 查看 Logs：发现数据库连接池耗尽
5. 根因：连接泄漏导致池耗尽

### 34.3.6 多租户系统的资源监控

在 SaaS 系统中，按租户维度统计资源使用情况，实现用量计费和公平调度。

### 34.4.2 Prometheus 中间件

```python
# app/monitoring/middleware.py
import time
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response
from app.monitoring.prometheus_metrics import (
    REQUEST_COUNT,
    REQUEST_DURATION,
    REQUEST_IN_PROGRESS,
)


class PrometheusMiddleware(BaseHTTPMiddleware):
    """
    自动采集每个 HTTP 请求的 Prometheus 指标
    """

    async def dispatch(self, request: Request, call_next) -> Response:
        method = request.method
        # 使用路由路径而非实际路径（避免高基数标签）
        path = self._get_route_path(request)

        # 跟踪进行中的请求
        REQUEST_IN_PROGRESS.labels(method=method, endpoint=path).inc()

        start_time = time.perf_counter()
        try:
            response = await call_next(request)
            status_code = response.status_code
        except Exception as exc:
            status_code = 500
            raise exc
        finally:
            duration = time.perf_counter() - start_time
            REQUEST_IN_PROGRESS.labels(method=method, endpoint=path).dec()
            REQUEST_COUNT.labels(
                method=method,
                endpoint=path,
                status_code=status_code,
            ).inc()
            REQUEST_DURATION.labels(
                method=method,
                endpoint=path,
            ).observe(duration)

        return response

    def _get_route_path(self, request: Request) -> str:
        """
        获取路由模板路径，避免高基数问题
        例如: /api/users/123 → /api/users/{user_id}
        """
        route = request.scope.get("route")
        if route and hasattr(route, "path"):
            return route.path
        return request.url.path
```

```python
from opentelemetry.exporter.prometheus import PrometheusMetricReader
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor
from opentelemetry.instrumentation.asyncpg import AsyncPGInstrumentor
from opentelemetry.instrumentation.redis import RedisInstrumentor


def setup_telemetry(
    app,
    service_name: str = "fastapi-app",
    service_version: str = "1.0.0",
    otlp_endpoint: str = "http://otel-collector:4317",
    enable_prometheus: bool = True,
    prometheus_port: int = 9090,
):
    """
    配置 OpenTelemetry 追踪和指标

    Args:
        app: FastAPI 应用实例
        service_name: 服务名称
        service_version: 服务版本
        otlp_endpoint: OTel Collector 的 gRPC 端点
        enable_prometheus: 是否启用 Prometheus 指标导出
        prometheus_port: Prometheus 指标导出端口
    """
    # 创建 Resource（描述产生遥测数据的实体）
    resource = Resource.create({
        SERVICE_NAME: service_name,
        SERVICE_VERSION: service_version,
        "deployment.environment": "production",
    })

    # ========================================
    # 配置 Tracing
    # ========================================
    trace_provider = TracerProvider(resource=resource)

    # OTLP Exporter（发送到 OTel Collector）
    otlp_span_exporter = OTLPSpanExporter(endpoint=otlp_endpoint, insecure=True)
    span_processor = BatchSpanProcessor(
        otlp_span_exporter,
        max_queue_size=2048,
        schedule_delay_millis=5000,
        max_export_batch_size=512,
        export_timeout_millis=30000,
    )
    trace_provider.add_span_processor(span_processor)
    trace.set_tracer_provider(trace_provider)

    # ========================================
    # 配置 Metrics
    # ========================================
    metric_readers = []

    # OTLP Metric Exporter
    otlp_metric_exporter = OTLPMetricExporter(endpoint=otlp_endpoint, insecure=True)
    otlp_metric_reader = PeriodicExportingMetricReader(
        exporter=otlp_metric_exporter,
        export_interval_millis=15000,  # 15 秒导出一次
    )
    metric_readers.append(otlp_metric_reader)

    # Prometheus Metric Reader（可选）
    if enable_prometheus:
        prometheus_reader = PrometheusMetricReader(port=prometheus_port)
        metric_readers.append(prometheus_reader)

    meter_provider = MeterProvider(
        resource=resource,
        metric_readers=metric_readers,
    )
    metrics.set_meter_provider(meter_provider)

    # ========================================
    # 自动仪表化（Auto-instrumentation）
    # ========================================
    # FastAPI
    FastAPIInstrumentor.instrument_app(app)

    # HTTP 客户端
    HTTPXClientInstrumentor().instrument()

    # 数据库
    AsyncPGInstrumentor().instrument()

    # Redis
    RedisInstrumentor().instrument()

    return {
        "tracer": trace.get_tracer(service_name, service_version),
        "meter": metrics.get_meter(service_name, service_version),
    }
```

```python
import logging
import structlog
from opentelemetry import trace
from opentelemetry import context as otel_context


def add_trace_info(logger, method_name, event_dict):
    """
    在日志中自动注入 OpenTelemetry Trace ID 和 Span ID
    """
    span = trace.get_current_span()
    if span and span.is_recording():
        ctx = span.get_span_context()
        event_dict["trace_id"] = format(ctx.trace_id, "032x")
        event_dict["span_id"] = format(ctx.span_id, "016x")
    return event_dict


def add_request_context(logger, method_name, event_dict):
    """
    添加请求上下文信息
    """
    # 从 contextvars 获取请求信息
    from app.core.context import get_request_id, get_user_id
    request_id = get_request_id()
    if request_id:
        event_dict["request_id"] = request_id
    user_id = get_user_id()
    if user_id:
        event_dict["user_id"] = user_id
    return event_dict


def setup_logging(log_level: str = "INFO", json_output: bool = True):
    """
    配置 structlog 结构化日志

    Args:
        log_level: 日志级别
        json_output: 是否输出 JSON 格式（生产环境用 JSON，开发环境用控制台格式）
    """
    shared_processors = [
        structlog.contextvars.merge_contextvars,
        structlog.stdlib.add_logger_name,
        structlog.stdlib.add_log_level,
        structlog.stdlib.PositionInfoRenderer(),
        add_trace_info,
        add_request_context,
        structlog.processors.StackInfoRenderer(),
        structlog.processors.format_exc_info,
        structlog.processors.TimeStamper(fmt="iso"),
    ]

    if json_output:
        # 生产环境: JSON 输出
        renderer = structlog.processors.JSONRenderer()
    else:
        # 开发环境: 彩色控制台输出
        renderer = structlog.dev.ConsoleRenderer()

    structlog.configure(
        processors=[
            *shared_processors,
            structlog.stdlib.ProcessorFormatter.wrap_for_formatter,
        ],
        logger_factory=structlog.stdlib.LoggerFactory(),
        wrapper_class=structlog.stdlib.BoundLogger,
        cache_logger_on_first_use=True,
    )

    # 配置标准库 logging 的格式化器
    formatter = structlog.stdlib.ProcessorFormatter(
        processors=[
            structlog.stdlib.ProcessorFormatter.remove_processors_meta,
            renderer,
        ],
        foreign_pre_chain=shared_processors,
    )

    handler = logging.StreamHandler()
    handler.setFormatter(formatter)

    root_logger = logging.getLogger()
    root_logger.handlers.clear()
    root_logger.addHandler(handler)
    root_logger.setLevel(getattr(logging, log_level.upper()))

    # 降低第三方库日志级别
    logging.getLogger("uvicorn.access").setLevel(logging.WARNING)
    logging.getLogger("uvicorn.error").setLevel(logging.INFO)


# 获取 logger 的便捷函数
def get_logger(name: str = __name__):
    return structlog.get_logger(name)
```

## 34.5 常见陷阱与最佳实践

### 34.5.1 陷阱：高基数标签导致 Prometheus 崩溃

**问题**：在 Prometheus 指标中使用用户 ID、请求 ID 等高基数（high-cardinality）值作为标签，导致时间序列数量爆炸。

```python
# 错误：user_id 是高基数标签（百万级用户 = 百万级时间序列）
REQUEST_COUNT.labels(user_id=str(user_id)).inc()

# 正确：使用低基数标签
REQUEST_COUNT.labels(method="GET", endpoint="/api/users").inc()
```

**原则**：标签值的唯一数量（基数）应远少于 100。HTTP 路径应使用路由模板（`/api/users/{id}`）而非实际路径（`/api/users/123`）。

### 34.5.2 陷阱：采样导致链路断裂

**问题**：使用 `TraceIdRatioBased` 采样器时，父 Span 被采样但子 Span 未被采样，导致链路断裂。

**解决方案**：始终使用 `ParentBased` 采样器包裹 `TraceIdRatioBased`，确保同一 Trace 内的采样决策一致：

```python
from opentelemetry.sdk.trace.sampling import ParentBased, TraceIdRatioBased

sampler = ParentBased(root=TraceIdRatioBased(rate=0.1))
trace_provider = TracerProvider(resource=resource, sampler=sampler)
```

### 34.5.3 陷阱：日志与追踪未关联

**问题**：日志和追踪是独立的，无法通过日志找到对应的 Trace，反之亦然。

**最佳实践**：在日志中自动注入 `trace_id` 和 `span_id`（见 34.4.5 节的 `add_trace_info` 处理器）。Grafana 和 Jaeger 都支持通过 trace_id 关联日志和追踪。

### 34.5.4 陷阱：健康检查依赖外部服务导致级联重启

**问题**：Liveness Probe 检查数据库连接。当数据库不可用时，所有实例被标记为不健康并重启，导致服务雪崩。

**最佳实践**：

- **Liveness Probe**：仅检查进程存活（内存不泄漏、不死锁）
- **Readiness Probe**：检查依赖服务（数据库、Redis）
- Readiness 失败 → 从负载均衡移除（不重启）
- Liveness 失败 → 重启容器

### 34.5.5 陷阱：Histogram bucket 配置不当

**问题**：默认 bucket 不适合 API 延迟分布。如果大部分请求在 10-50ms，但默认 bucket 在 0.05-0.1 处，P99 计算精度差。

**最佳实践**：根据实际延迟分布选择 bucket 边界：

```python
# API 服务（延迟在毫秒级）
buckets=[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0, 10.0]

# 数据库查询（延迟在亚毫秒级）
buckets=[0.001, 0.005, 0.01, 0.05, 0.1, 0.5, 1.0]
```

### 34.5.6 陷阱：OpenTelemetry 性能开销

**问题**：全量采集（AlwaysOn 采样）在高流量下显著增加 CPU 和内存使用，且 Exporter 可能成为瓶颈。

**最佳实践**：

1. 使用 `BatchSpanProcessor` 而非 `SimpleSpanProcessor`
2. 生产环境使用 10% 采样率 + 错误/慢请求全量采集
3. 监控 OTel SDK 自身的指标（队列大小、导出延迟）
4. 设置 Exporter 超时，避免阻塞应用

### 34.5.7 陷阱：日志输出到文件而非 stdout

**问题**：容器中将日志写入文件而非 stdout/stderr，导致 Docker/K8s 日志驱动无法收集。

**最佳实践**：容器化应用应始终将日志输出到 stdout/stderr，由日志收集代理（Fluentd, Promtail）统一采集。

### 34.5.8 最佳实践清单

1. **区分 Liveness 和 Readiness 探针**：避免级联重启
2. **使用低基数标签**：避免 Prometheus 时间序列爆炸
3. **结构化日志 + Trace ID**：实现日志与追踪关联
4. **使用 ParentBased 采样**：保持链路完整性
5. **选择合适的 Histogram bucket**：匹配实际延迟分布
6. **使用 BatchSpanProcessor**：减少导出开销
7. **尾部采样**：仅保留错误和慢请求的完整链路
8. **监控监控系统**：采集 OTel SDK 和 Collector 自身指标
9. **设置告警**：错误率、延迟、可用性三维度
10. **使用路由路径而非实际路径**：作为 Prometheus 标签
11. **配置日志级别**：生产环境 INFO，第三方库 WARNING 以上
