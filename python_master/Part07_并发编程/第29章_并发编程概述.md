# 第29章 并发编程概述

## 1. 概念与语法

### 1.1 并发 vs 并行

**并发 (Concurrency)** 和 **并行 (Parallelism)** 是理解 Python 并发编程的基石，两者经常被混淆，但本质截然不同。

**并发**是指多个任务在**逻辑上**同时推进的能力。它强调的是**结构**——程序可以同时处理多个独立的事务，但这些事务不一定在同一物理时刻执行。操作系统通过时间片轮转 (time slicing) 让多个线程交替执行，营造出"同时"的假象。

**并行**是指多个任务在**物理上**同时执行的能力。它要求硬件提供多个处理核心，每个核心独立执行各自的指令流。

```
并发 (单核时间片轮转):
Core 0: [Task A][Task B][Task A][Task B][Task A]
         |----时间片----|----时间片----|----时间片----|

并行 (多核同时执行):
Core 0: [Task A][Task A][Task A][Task A][Task A]
Core 1: [Task B][Task B][Task B][Task B][Task B]
Core 2: [Task C][Task C][Task C][Task C][Task C]
```

Rob Pike 的经典定义：

> 并发是关于**处理**多件事，并行是关于**做**多件事。

用更精确的术语表述：

| 维度 | 并发 (Concurrency) | 并行 (Parallelism) |
|------|---------------------|---------------------|
| 本质 | 结构属性 | 执行属性 |
| 核心要求 | 任务可交替推进 | 多个物理执行单元 |
| 目标 | 提高吞吐量/响应性 | 提高计算速度 |
| 关系 | 并行是并发的特殊形式 | 并行一定并发，并发不一定并行 |

### 1.2 CPU 密集型 vs IO 密集型

任务是 **CPU-bound** 还是 **IO-bound**，直接决定了并发模型的选择。

**CPU 密集型 (CPU-bound)**：任务的瓶颈在于 CPU 计算能力。计算密集的数值运算、加密哈希、图像处理、矩阵运算等均属此类。

```python
# CPU密集型示例：质数判定
def is_prime(n):
    if n < 2:
        return False
    for i in range(2, int(n ** 0.5) + 1):
        if n % i == 0:
            return False
    return True

# 这类任务几乎不涉及IO等待，CPU全速运转
result = sum(1 for i in range(10_000_000) if is_prime(i))
```

**IO 密集型 (IO-bound)**：任务的瓶颈在于等待外部 IO 操作完成。网络请求、磁盘读写、数据库查询等均属此类。

```python
# IO密集型示例：批量HTTP请求
import urllib.request

def fetch_url(url):
    """大部分时间在等待网络响应，CPU几乎空闲"""
    response = urllib.request.urlopen(url)
    return response.read()

# CPU利用率极低，线程在等待IO时释放GIL
urls = [f"https://httpbin.org/delay/1" for _ in range(10)]
```

关键区别在**资源等待模式**：

```
CPU密集型任务时间线 (CPU始终忙碌):
[====计算====][====计算====][====计算====]

IO密集型任务时间线 (CPU大部分时间在等待):
[==发起IO==]........等待........[==处理响应==]
             ^^^^^^^^^^^^^^^^^^^^
             这段时间CPU可以做其他工作
```

### 1.3 Python 并发模型概览

Python 提供三大并发模型，各有适用场景：

#### 1.3.1 多线程 (threading)

基于操作系统原生线程的并发模型。Python 线程是真正的 OS 级线程，由操作系统调度。但由于 GIL 的存在，同一时刻只有一个线程执行 Python 字节码。

```python
import threading

def worker(name, count):
    for _ in range(count):
        pass  # Python字节码执行
    print(f"{name} done")

t1 = threading.Thread(target=worker, args=("Thread-1", 1000000))
t2 = threading.Thread(target=worker, args=("Thread-2", 1000000))
t1.start(); t2.start()
t1.join(); t2.join()
```

#### 1.3.2 多进程 (multiprocessing)

每个进程拥有独立的 Python 解释器和 GIL，可以真正实现并行计算。适用于 CPU 密集型任务。

```python
from multiprocessing import Process

def worker(name, count):
    for _ in range(count):
        pass
    print(f"{name} done")

p1 = Process(target=worker, args=("Process-1", 1000000))
p2 = Process(target=worker, args=("Process-2", 1000000))
p1.start(); p2.start()
p1.join(); p2.join()
```

#### 1.3.3 协程 (asyncio)

基于事件循环的异步并发模型。单线程内通过协程 (coroutine) 实现协作式调度，避免了线程切换的开销和锁的复杂性。

```python
import asyncio

async def worker(name, seconds):
    await asyncio.sleep(seconds)  # 非阻塞等待
    print(f"{name} done after {seconds}s")

async def main():
    # 三个协程并发执行，总耗时约2秒而非3秒
    await asyncio.gather(
        worker("Coroutine-1", 1),
        worker("Coroutine-2", 2),
        worker("Coroutine-3", 1.5),
    )

asyncio.run(main())
```

### 1.4 全局解释器锁 (GIL) 简介

**GIL (Global Interpreter Lock)** 是 CPython 解释器中最具争议的设计之一。它是一把互斥锁 (mutex)，保护着 Python 对象的引用计数机制，确保同一时刻只有一个线程执行 Python 字节码。

GIL 的存在使得：

- **多线程无法利用多核**：即使有多个线程，也只有一个线程在执行 Python 代码
- **IO 操作自动释放 GIL**：线程在执行 IO 操作前会主动释放 GIL，等待完成后重新获取
- **C 扩展可手动释放 GIL**：如 NumPy 在执行底层 C 计算时释放 GIL

```python
# GIL的直观体现
import threading, time

def cpu_task():
    """纯Python计算——受GIL限制，多线程不会加速"""
    total = 0
    for i in range(50_000_000):
        total += i

# 单线程耗时约4秒
start = time.perf_counter()
cpu_task()
cpu_task()
print(f"单线程: {time.perf_counter() - start:.2f}s")

# 双线程耗时也约4秒——GIL导致无法真正并行
start = time.perf_counter()
t1 = threading.Thread(target=cpu_task)
t2 = threading.Thread(target=cpu_task)
t1.start(); t2.start()
t1.join(); t2.join()
print(f"双线程: {time.perf_counter() - start:.2f}s")
```

### 1.5 并发选择决策树

```
你的任务类型是什么？
├── CPU密集型 (计算、数据处理)
│   ├── 需要利用多核？
│   │   ├── 是 → multiprocessing
│   │   └── 否 → 单线程即可（避免多线程的开销）
│   └── 任务可分片？
│       ├── 是 → multiprocessing.Pool + map
│       └── 否 → concurrent.futures.ProcessPoolExecutor
├── IO密集型 (网络、文件、数据库)
│   ├── 大量并发连接 (1000+)
│   │   ├── 是 → asyncio
│   │   └── 否 → threading / concurrent.futures.ThreadPoolExecutor
│   └── 需要与同步库交互？
│       ├── 是 → threading (或 asyncio + run_in_executor)
│       └── 否 → asyncio (原生异步库)
└── 混合型 (CPU + IO)
    ├── IO为主 → asyncio + run_in_executor 处理CPU部分
    └── CPU为主 → multiprocessing + 进程间通信
```

更详细的决策矩阵：

| 条件 | 推荐 | 原因 |
|------|------|------|
| CPU 密集型 + 需要并行 | multiprocessing | 绕过 GIL，真正多核并行 |
| CPU 密集型 + 少量任务 | ProcessPoolExecutor | 简洁 API，进程池管理 |
| IO 密集型 + 少量并发 (10-100) | threading | 简单直观，开销可接受 |
| IO 密集型 + 大量并发 (1000+) | asyncio | 单线程事件循环，极低开销 |
| IO 密集型 + 需要混合同步/异步 | asyncio + ThreadPoolExecutor | 异步为主，同步任务放入线程池 |
| 混合型 + 高性能 | multiprocessing + asyncio | 进程级并行 + 协程级并发 |

## 2. 原理与机制

### 2.1 操作系统层面的并发基础

理解 Python 并发必须先理解操作系统的并发原语。

#### 2.1.1 进程 (Process)

进程是操作系统资源分配的基本单位。每个进程拥有独立的：

- **虚拟地址空间**：代码段、数据段、堆、栈
- **文件描述符表**：打开的文件、socket
- **信号处理机制**：注册的信号处理器
- **进程控制块 (PCB)**：PID、状态、优先级、CPU 寄存器快照

在 Linux 中，进程通过 `fork()` 系统调用创建：

```c
// Linux kernel: kernel/fork.c (简化)
// copy_process() 创建新进程的核心逻辑
static __latent_entropy struct task_struct *copy_process(
    struct pid *pid, int trace, int node)
{
    // 分配新的 task_struct
    p = dup_task_struct(current, node);
    
    // 复制进程上下文
    retval = copy_mm(clone_flags, p);      // 复制内存空间(COW)
    retval = copy_files(clone_flags, p);    // 复制文件描述符表
    retval = copy_fs(clone_flags, p);       // 复制文件系统信息
    retval = copy_signal(clone_flags, p);   // 复制信号处理
    retval = copy_sighand(clone_flags, p);  // 复制信号处理器
    
    // 设置PID和父子关系
    p->pid = pid_nr(pid);
    p->real_parent = current;
    
    return p;
}
```

`fork()` 采用 **写时复制 (Copy-On-Write, COW)** 策略：父子进程初始共享物理内存页，仅当某方修改页面时才复制，极大降低了进程创建开销。

#### 2.1.2 线程 (Thread)

线程是 CPU 调度的基本单位。同一进程内的线程共享：

- 虚拟地址空间（代码段、数据段、堆）
- 文件描述符表
- 信号处理机制

但每个线程有独立的：

- 栈空间
- 程序计数器 (PC)
- 寄存器集合
- 线程局部存储 (TLS)

Linux 中线程通过 `clone()` 系统调用创建，与进程共享大量资源：

```c
// Linux kernel: kernel/fork.c
// 创建线程时共享地址空间、文件系统、文件描述符、信号处理器
clone_flags = CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND;
// 创建进程时仅共享文件系统等，不共享地址空间
clone_flags = CLONE_FS | CLONE_FILES | CLONE_SIGHAND;
```

#### 2.1.3 上下文切换开销对比

| 操作 | 大致耗时 | 说明 |
|------|----------|------|
| 线程上下文切换 | ~1-10 μs | 切换寄存器、栈指针、TLB 刷新 |
| 进程上下文切换 | ~5-50 μs | 额外切换地址空间、TLB 全量刷新 |
| 协程上下文切换 | ~0.1-1 μs | 用户态切换，无内核参与 |

### 2.2 Python 进程与线程的 CPython 实现

#### 2.2.1 Python 线程的创建

CPython 的 `threading.Thread` 底层调用操作系统线程 API。以 Linux 为例：

```c
// Modules/_threadmodule.c
// 创建线程的底层实现
static PyObject *
thread_PyThread_start_new_thread(PyObject *self, PyObject *fargs)
{
    // 分配引导参数
    boot = PyMem_NEW(struct bootstate, 1);
    boot->func = func;      // 线程目标函数
    boot->args = args;      // 位置参数
    boot->keyw = keyw;     // 关键字参数
    
    // 调用操作系统线程创建API
    // Linux: pthread_create
    // Windows: _beginthreadex
    ident = PyThread_start_new_thread(t_bootstrap, (void*)boot);
    
    return ident;
}

// 线程引导函数
static void t_bootstrap(void *boot_raw)
{
    struct bootstate *boot = (struct bootstate *)boot_raw;
    
    // 获取GIL
    PyGILState_STATE gstate = PyGILState_Ensure();
    
    // 执行目标函数
    PyObject *res = PyObject_Call(boot->func, boot->args, boot->keyw);
    
    // 释放GIL，清理资源
    PyGILState_Release(gstate);
}
```

#### 2.2.2 Python 进程的创建

`multiprocessing` 模块在不同平台使用不同策略创建子进程：

```c
// Python进程创建的底层路径 (Linux fork模式)
// multiprocessing/process.py → _fork() → os.fork()

// os.fork() 最终调用:
// Lib/os.py → C层 posix_fork_impl()
// Modules/posixmodule.c
static PyObject *
os_fork_impl(PyObject *module)
{
    pid_t pid;
    // 调用Linux fork系统调用
    pid = fork();
    if (pid == 0) {
        // 子进程：重新初始化随机数种子
        // 重置文件锁、线程状态等
    }
    return pid;
}
```

### 2.3 GIL 的 CPython 源码级分析

#### 2.3.1 GIL 的数据结构

GIL 在 CPython 3.12+ 中的实现位于 `Python/ceval_gil.c`：

```c
// Python/ceval_gil.c (Python 3.12+)
// GIL的底层数据结构
struct _gil_runtime_state {
    unsigned long interval;       // GIL检查间隔(微秒)
    _Py_atomic_int locked;        // GIL是否被持有
    unsigned long switch_number;   // GIL切换次数
    _Py_atomic_int eval_breaker;  // 中断标志，触发GIL释放检查
    
    // 条件变量和互斥锁
    PyCOND_T cond;                // GIL可用条件变量
    PyMUTEX_T mutex;              // 保护GIL状态的互斥锁
    
    // 等待GIL的线程计数
    _Py_atomic_int waiters;       // 等待获取GIL的线程数
};
```

#### 2.3.2 GIL 的获取与释放

```c
// Python/ceval_gil.c
// 获取GIL
static void take_gil(PyThreadState *tstate)
{
    struct _gil_runtime_state *gil = &tstate->interp->ceval.gil;
    
    // 1. 加锁保护GIL状态
    PyMUTEX_LOCK(&gil->mutex);
    
    // 2. 如果GIL被其他线程持有，等待
    while (_Py_atomic_load_int(&gil->locked)) {
        // 增加等待者计数
        _Py_atomic_add_int(&gil->waiters, 1);
        
        // 等待条件变量通知
        PyCOND_WAIT(&gil->cond, &gil->mutex);
        
        // 减少等待者计数
        _Py_atomic_add_int(&gil->waiters, -1);
    }
    
    // 3. 获取GIL
    _Py_atomic_store_int(&gil->locked, 1);
    
    // 4. 解锁
    PyMUTEX_UNLOCK(&gil->mutex);
}
```

```c
// 释放GIL
static void drop_gil(PyThreadState *tstate)
{
    struct _gil_runtime_state *gil = &tstate->interp->ceval.gil;
    
    // 1. 加锁
    PyMUTEX_LOCK(&gil->mutex);
    
    // 2. 释放GIL
    _Py_atomic_store_int(&gil->locked, 0);
    
    // 3. 如果有线程在等待，通知它们
    if (_Py_atomic_load_int(&gil->waiters)) {
        PyCOND_SIGNAL(&gil->cond);
    }
    
    // 4. 解锁
    PyMUTEX_UNLOCK(&gil->mutex);
}
```

#### 2.3.3 GIL 的检查间隔机制

在 Python 3.2+ (Antoine Pitrou 的 GIL 重构) 中，GIL 的切换基于**检查间隔 (check interval)**：

```c
// Python/ceval.c
// 主解释器循环中的GIL检查
PyObject *_PyEval_EvalFrameDefault(PyThreadState *tstate, 
                                     PyFrameObject *f, int throwflag)
{
    // ... 主执行循环 ...
    for (;;) {
        // 检查是否需要释放GIL
        // eval_breaker是一个原子变量，当其他线程请求GIL时被设置
        if (_Py_atomic_load_int(&gil->eval_breaker)) {
            // 主动让出GIL给等待的线程
            drop_gil(tstate);
            take_gil(tstate);
        }
        
        // 执行一条字节码指令
        opcode = NEXTOP();
        // ... 指令分派和执行 ...
    }
}
```

Python 3.12 中引入了基于时间的 GIL 切换机制 (`PEP 703` 前的优化)：

```c
// Python/ceval_gil.c
// 设置GIL切换间隔
int _PyGilState_SetCheckInterval(int microseconds)
{
    // 将微秒转换为评估步数
    // 默认值5000 (5毫秒)
    struct _gil_runtime_state *gil = &tstate->interp->ceval.gil;
    gil->interval = microseconds;
    return 0;
}
```

#### 2.3.4 GIL 的存在原因

GIL 的根本目的是保护 CPython 的**引用计数机制**：

```c
// Include/object.h
// Python对象的引用计数操作
#define Py_INCREF(op) (                          \
    _Py_INC_REFPTR((PyObject *)(op)))            \

#define _Py_INC_REFPTR(op) (++((op)->ob_refcnt))

// 如果没有GIL，多线程同时修改ob_refcnt会导致竞态条件
// 即使使用原子操作，也需要保护所有涉及引用计数的操作链
typedef struct _object {
    Py_ssize_t ob_refcnt;     // 引用计数 ← 需要GIL保护
    PyTypeObject *ob_type;    // 类型指针
} PyObject;
```

移除 GIL 的困难在于：

1. **引用计数的原子性**：每个 Python 对象都有引用计数，几乎每个操作都会修改引用计数。将所有引用计数操作改为原子操作会带来 2-10 倍的性能下降（单线程场景）。
2. **内部数据结构保护**：dict、list 等内置类型的实现没有细粒度锁，GIL 是唯一的保护。
3. **C 扩展兼容性**：大量 C 扩展依赖 GIL 保证线程安全，移除 GIL 意味着所有 C 扩展需要重写。
4. **性能权衡**：细粒度锁方案在单线程场景下比 GIL 慢 10-30%。

### 3.1 多线程适用场景

**场景 1：网络 IO 并发**

Web 爬虫、API 调用、数据库查询等网络密集型操作。

```python
import threading
import urllib.request
from time import perf_counter

def fetch(url):
    """网络请求——线程等待IO时自动释放GIL"""
    try:
        response = urllib.request.urlopen(url, timeout=10)
        return len(response.read())
    except Exception as e:
        return str(e)

def threaded_fetch(urls, max_workers=10):
    """多线程并发抓取"""
    results = [None] * len(urls)
    
    def worker(idx, url):
        results[idx] = fetch(url)
    
    threads = []
    for i, url in enumerate(urls):
        t = threading.Thread(target=worker, args=(i, url))
        threads.append(t)
        t.start()
    
    for t in threads:
        t.join()
    
    return results

# 10个请求并发：总耗时≈单次请求耗时，而非10倍
urls = [f"https://httpbin.org/delay/1" for _ in range(10)]
start = perf_counter()
results = threaded_fetch(urls)
print(f"总耗时: {perf_counter() - start:.2f}s")  # ≈1-2s，而非10s
```

```python
对比串行、多线程、多进程、协程在不同任务类型下的表现

import threading
import multiprocessing as mp
import asyncio
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# ============ CPU密集型任务 ============
def cpu_task(n):
    """纯Python计算——受GIL限制"""
    total = 0
    for i in range(n):
        total += i
    return total

# ============ IO密集型任务 ============
def io_task(seconds):
    """模拟IO等待"""
    time.sleep(seconds)
    return seconds

async def async_io_task(seconds):
    """异步IO等待"""
    await asyncio.sleep(seconds)
    return seconds

def benchmark_cpu():
    """CPU密集型基准测试"""
    N = 5_000_000
    TASK_COUNT = 4
    
    print("=" * 60)
    print("CPU密集型任务基准测试")
```

### 4.2 GIL 行为验证实验

```python
"""
GIL行为验证实验
直观展示GIL对不同类型任务的影响
"""
import time
import threading
import multiprocessing as mp

def cpu_bound_task(name, iterations):
    """CPU密集型：纯Python计算"""
    total = 0
    for i in range(iterations):
        total += i
    print(f"  {name}: 结果={total}")
    return total

def mixed_task(name, iterations, sleep_time):
    """混合型：计算+IO"""
    total = 0
    for i in range(iterations):
        total += i
        if i % (iterations // 5) == 0:
            time.sleep(sleep_time)  # IO等待时释放GIL
    print(f"  {name}: 结果={total}")
    return total

def experiment_cpu_threads():
    """实验1：CPU密集型任务的多线程无加速"""
    print("\n[实验1] CPU密集型任务——多线程无法利用多核")
    N = 10_000_000
    
    # 单线程执行2个任务
    start = time.perf_counter()
    cpu_bound_task("单线程-1", N)
    cpu_bound_task("单线程-2", N)
    single = time.perf_counter() - start
    print(f"  单线程总耗时: {single:.3f}s")
    
    # 双线程执行2个任务
    start = time.perf_counter()
    t1 = threading.Thread(target=cpu_bound_task, args=("线程-1", N))
    t2 = threading.Thread(target=cpu_bound_task, args=("线程-2", N))
    t1.start(); t2.start()
    t1.join(); t2.join()
    multi = time.perf_counter() - start
    print(f"  双线程总耗时: {multi:.3f}s")
    print(f"  结论: 双线程比单线程更慢(GIL导致的频繁切换开销)")

def experiment_cpu_processes():
    """实验2：CPU密集型任务的多进程真正并行"""
    print("\n[实验2] CPU密集型任务——多进程真正并行")
    N = 10_000_000
    
    # 双进程执行2个任务
    start = time.perf_counter()
    p1 = mp.Process(target=cpu_bound_task, args=("进程-1", N))
    p2 = mp.Process(target=cpu_bound_task, args=("进程-2", N))
    p1.start(); p2.start()
    p1.join(); p2.join()
    multi = time.perf_counter() - start
    print(f"  双进程总耗时: {multi:.3f}s")
    print(f"  结论: 多进程绕过GIL，实现真正并行")

def experiment_io_threads():
    """实验3：IO密集型任务的多线程有效"""
    print("\n[实验3] IO密集型任务——多线程有效加速")
    
    # 单线程执行2个任务
    start = time.perf_counter()
    mixed_task("单线程-1", 100_000, 0.5)
    mixed_task("单线程-2", 100_000, 0.5)
    single = time.perf_counter() - start
    print(f"  单线程总耗时: {single:.3f}s")
    
    # 双线程执行2个任务
    start = time.perf_counter()

    t1.start(); t2.start()
    t1.join(); t2.join()
    multi = time.perf_counter() - start
    print(f"  双线程总耗时: {multi:.3f}s")
    print(f"  结论: IO等待时释放GIL，其他线程可执行")

if __name__ == "__main__":
```

### 4.3 并发选择决策树实现

```python
"""
并发模型选择决策辅助工具
根据任务特征推荐最优并发模型
"""
from dataclasses import dataclass
from enum import Enum
from typing import List

class ConcurrencyModel(Enum):
    SERIAL = "串行（无并发）"
    THREADING = "多线程 (threading)"
    MULTIPROCESSING = "多进程 (multiprocessing)"
    ASYNCIO = "异步协程 (asyncio)"
    HYBRID_THREAD = "混合模型 (asyncio + ThreadPoolExecutor)"
    HYBRID_PROCESS = "混合模型 (asyncio + ProcessPoolExecutor)"

@dataclass
class TaskProfile:
    """任务画像"""
    is_cpu_bound: bool          # 是否CPU密集
    is_io_bound: bool          # 是否IO密集
    io_type: str = "network"   # IO类型: network/file/database
    concurrent_count: int = 1  # 并发数量
    need_shared_state: bool = False  # 是否需要共享状态
    need_realtime: bool = False      # 是否需要实时响应
    has_native_async: bool = False   # 是否有原生异步库
    cpu_cores: int = 4               # 可用CPU核心数

def recommend_concurrency(profile: TaskProfile) -> List[str]:
    """根据任务画像推荐并发模型"""
    recommendations = []
    
    # CPU密集型
    if profile.is_cpu_bound and not profile.is_io_bound:
        if profile.concurrent_count == 1:
            recommendations.append(f"→ {ConcurrencyModel.SERIAL.value}")
            recommendations.append("  原因: 单任务无需并发")
        elif profile.concurrent_count <= profile.cpu_cores:
            recommendations.append(f"→ {ConcurrencyModel.MULTIPROCESSING.value}")
            recommendations.append("  原因: CPU密集型需要绕过GIL，多进程实现真正并行")
            recommendations.append(f"  建议: 使用 ProcessPoolExecutor, workers={profile.concurrent_count}")
        else:
            recommendations.append(f"→ {ConcurrencyModel.MULTIPROCESSING.value}")
            recommendations.append(f"  原因: CPU密集型多进程并行，进程数建议等于核心数({profile.cpu_cores})")
            recommendations.append(f"  建议: 使用进程池，避免创建过多进程")
    
    # IO密集型
    elif profile.is_io_bound and not profile.is_cpu_bound:
        if profile.concurrent_count <= 10:
            recommendations.append(f"→ {ConcurrencyModel.THREADING.value}")
            recommendations.append("  原因: 少量并发，多线程简单直观")
        elif profile.concurrent_count <= 100:
            if profile.has_native_async:
                recommendations.append(f"→ {ConcurrencyModel.ASYNCIO.value}")
                recommendations.append("  原因: 中等并发+有原生异步库，asyncio更高效")
            else:
                recommendations.append(f"→ {ConcurrencyModel.THREADING.value}")
                recommendations.append("  原因: 中等并发但无原生异步库，多线程+同步库更实用")
        else:
            recommendations.append(f"→ {ConcurrencyModel.ASYNCIO.value}")
            recommendations.append("  原因: 大规模并发(1000+)，asyncio单线程事件循环最节省资源")
    
    # 混合型
    elif profile.is_cpu_bound and profile.is_io_bound:
        recommendations.append(f"→ {ConcurrencyModel.HYBRID_PROCESS.value}")
        recommendations.append("  原因: 混合型任务，asyncio处理IO，进程池处理CPU计算")
        recommendations.append("  方案: asyncio.run_in_executor(ProcessPoolExecutor)")
    
    # 需要共享状态
    if profile.need_shared_state and not profile.is_cpu_bound:
        recommendations.append("  补充: 如需共享状态，优先考虑 threading + 锁")
    
    return recommendations

def print_decision_guide():
    """打印完整决策指南"""
    print("=" * 70)
    print("Python 并发模型选择决策指南")
    print("=" * 70)
    
    scenarios = [
        TaskProfile(is_cpu_bound=True, is_io_bound=False,
                    concurrent_count=1, cpu_cores=4),
        TaskProfile(is_cpu_bound=True, is_io_bound=False,
                    concurrent_count=4, cpu_cores=4),
        TaskProfile(is_cpu_bound=True, is_io_bound=False,
                    concurrent_count=100, cpu_cores=4),
        TaskProfile(is_io_bound=True, is_cpu_bound=False,
                    concurrent_count=5, io_type="network"),
        TaskProfile(is_io_bound=True, is_cpu_bound=False,
                    concurrent_count=50, io_type="network", has_native_async=True),
        TaskProfile(is_io_bound=True, is_cpu_bound=False,
                    concurrent_count=5000, io_type="network", has_native_async=True),
        TaskProfile(is_cpu_bound=True, is_io_bound=True,
                    concurrent_count=10, cpu_cores=4),
    ]
    
    labels = [
        "单任务CPU密集",
        "4任务CPU密集(4核)",
        "100任务CPU密集(4核)",
        "5任务IO密集(网络)",
        "50任务IO密集(网络,有异步库)",
        "5000任务IO密集(网络,有异步库)",
        "混合型(CPU+IO)",
    ]
    
    for label, profile in zip(labels, scenarios):
        print(f"\n场景: {label}")
        recs = recommend_concurrency(profile)
        for line in recs:
            print(f"  {line}")

if __name__ == "__main__":
    print_decision_guide()
```

### 5.1 陷阱：误用多线程处理 CPU 密集型任务

```python
# ❌ 错误：用多线程加速CPU密集型计算——GIL阻止了并行
```

### 5.2 陷阱：在 asyncio 中调用阻塞函数

```python
import asyncio

# ❌ 错误：在协程中调用阻塞函数——阻塞整个事件循环
```

### 5.3 陷阱：进程间共享可变状态

```python
# ❌ 错误：进程间不能直接共享Python对象
from multiprocessing import Process

shared_list = []  # 主进程中的列表

def worker():
    shared_list.append(1)  # 这修改的是子进程的副本！
    print(f"子进程: {shared_list}")  # [1]

p = Process(target=worker)
p.start()
p.join()
print(f"主进程: {shared_list}")  # [] ——子进程的修改不可见！

# ✅ 正确：使用Manager共享状态
from multiprocessing import Manager

def worker(shared_list):
    shared_list.append(1)  # 通过Manager代理修改

with Manager() as manager:
    shared_list = manager.list()  # 代理列表
    p = Process(target=worker, args=(shared_list,))
    p.start()
    p.join()
    print(f"主进程: {list(shared_list)}")  # [1] ——修改可见！
```

### 5.4 陷阱：守护线程中的资源泄漏

```python
import threading
import time

# ❌ 错误：守护线程可能在中途被强制终止
def worker():
    with open("important.log", "w") as f:
        for i in range(10):
            f.write(f"Line {i}\n")
            time.sleep(1)  # 如果主线程退出，文件可能未正确关闭！

t = threading.Thread(target=worker, daemon=True)  # 危险！
t.start()
time.sleep(3)  # 3秒后主线程退出，worker被强制终止
# 文件可能只写了3行，且可能未flush

# ✅ 正确：确保资源正确清理
def worker_safe(stop_event):
    with open("important.log", "w") as f:
        for i in range(10):
            if stop_event.is_set():
                break  # 优雅退出
            f.write(f"Line {i}\n")
            f.flush()
    print("Worker cleanly exited")

stop_event = threading.Event()
t = threading.Thread(target=worker_safe, args=(stop_event,))
t.daemon = False  # 非守护线程，join确保等待完成
t.start()
time.sleep(3)
stop_event.set()  # 通知线程退出
t.join(timeout=5)  # 等待线程完成
```

### 5.5 陷阱：全局解释器锁检查间隔误用

```python
import sys

# ❌ 错误：增大检查间隔可能导致线程饥饿
sys.setswitchinterval(5.0)  # 5秒才切换一次线程！
# IO线程可能等待很久才能获得GIL

# ❌ 错误：过小的检查间隔导致频繁上下文切换
sys.setswitchinterval(0.000001)  # 1微秒切换一次
# 大量时间浪费在上下文切换上

# ✅ 正确：使用合理默认值或根据场景微调
# Python 3.12+ 默认: 5ms (0.005s)
# 通常不需要修改，除非有明确性能测试依据
print(f"当前切换间隔: {sys.getswitchinterval()}s")  # 0.005
```

### 5.6 最佳实践总结

| 实践 | 说明 |
|------|------|
| **CPU 密集型用多进程** | 绕过 GIL，真正利用多核 |
| **IO 密集型用多线程或协程** | 少量并发用线程，大量并发用协程 |
| **避免在协程中阻塞** | 使用 `await` 或 `run_in_executor` |
| **进程间避免共享状态** | 使用 Queue/Pipe 通信，而非共享内存 |
| **守护线程慎用** | 确保资源清理，优先使用 Event 信号退出 |
| **不要过度并发** | 线程数/进程数不超过核心数的 2-5 倍（IO 密集型可更多） |
| **优先使用高层 API** | `concurrent.futures` 优于手动管理线程/进程 |
| **测量再优化** | 先测量瓶颈再选择并发模型，避免过早优化 |
| **关注启动开销** | 进程创建开销远大于线程，短任务不适合多进程 |
| **混合模型需谨慎** | asyncio + 进程池/线程池的组合要确保资源正确释放 |

### 5.7 性能对比速查表

| 模型 | 创建开销 | 内存占用 | 并发上限 | GIL 影响 | 适用场景 |
|------|----------|----------|----------|----------|----------|
| threading | ~1ms | ~8MB/线程 | 数千 | CPU 受限 | IO 密集型 |
| multiprocessing | ~50ms | ~30MB/进程 | 数十 | 无 | CPU 密集型 |
| asyncio | ~1μs | ~1KB/协程 | 数万 | 无 | 大规模 IO |

> **提示**：这些数值是数量级估计，实际值取决于操作系统、硬件和 Python 版本。始终在实际环境中测量验证。
