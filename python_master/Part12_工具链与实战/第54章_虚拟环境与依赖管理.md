# 第54章 虚拟环境与依赖管理

虚拟环境是 Python 项目隔离的基础设施。本章从 venv 的底层实现、多种依赖管理工具的对比到生产环境最佳实践，深入剖析虚拟环境的工作机制。

---

## 54.1 概念与语法

### 54.1.1 venv 标准库方案

```bash
# 创建虚拟环境
python -m venv myproject_env

# 激活
# macOS/Linux:
source myproject_env/bin/activate
# Windows:
myproject_env\Scripts\activate

# 停用
deactivate

# 指定 Python 版本创建
python3.12 -m venv myproject_env

# 创建时不安装 pip（最小化）
python -m venv --without-pip myproject_env

# 指定 prompt 名称
python -m venv --prompt=myproject myproject_env
```

### 54.1.2 虚拟环境的目录结构

```text
myproject_env/
├── bin/                    # macOS/Linux
│   ├── python              → ../lib/python3.12 （符号链接）
│   ├── python3             → python
│   ├── pip
│   ├── activate            # 激活脚本
│   └── ...
├── Scripts/                # Windows
│   ├── python.exe
│   ├── pip.exe
│   ├── activate.bat
│   └── Activate.ps1
├── lib/
│   └── python3.12/
│       └── site-packages/  # 第三方包安装位置
├── pyvenv.cfg              # 虚拟环境配置
└── include/                # C 扩展头文件
```

```ini
# pyvenv.cfg 内容
home = /usr/bin
include-system-site-packages = false
version = 3.12.0
prompt = myproject
```

### 54.1.3 依赖管理工具对比

| 工具 | 虚拟环境 | 依赖声明 | 锁文件 | 构建后端 | 亮点 |
|------|---------|---------|--------|---------|------|
| pip + venv | 手动 | requirements.txt | 无 | setuptools | 最基础 |
| pip-tools | 手动 | requirements.in | requirements.txt | setuptools | 精确锁定 |
| Poetry | 自动 | pyproject.toml | poetry.lock | poetry-core | 全功能 |
| PDM | 自动 | pyproject.toml | pdm.lock | 后端可选 | PEP 582 |
| uv | 自动 | pyproject.toml | uv.lock | 后端可选 | 极速 Rust 实现 |
| Conda | 自动 | environment.yml | 无 | 自有 | 数据科学 |
| Hatch | 自动 | pyproject.toml | 无 | hatchling | 环境矩阵 |

### 54.1.4 Poetry 工作流

```bash
# 安装
pip install poetry

# 初始化项目
poetry new myproject
# 或在现有项目中
poetry init

# 添加依赖
poetry add requests
poetry add --dev pytest

# 安装依赖
poetry install

# 运行命令
poetry run python script.py
poetry run pytest

# 构建与发布
poetry build
poetry publish

# 虚拟环境管理
poetry env info
poetry env list
poetry shell
```

## 54.2 原理与机制

### 54.2.1 venv 的底层实现

```python
# python -m venv 执行流程：
# 1. 创建目标目录
# 2. 复制或符号链接 Python 解释器
#    Unix: bin/python → 真实 python 的符号链接
#    Windows: 复制 python.exe + python312.dll
# 3. 创建 pyvenv.cfg 配置文件
# 4. 安装 pip（从 ensurepip 模块）
# 5. 创建 site-packages 目录

# 激活脚本的本质（Unix）：
# activate 脚本修改了以下环境变量：
# 1. VIRTUAL_ENV = /path/to/myproject_env
# 2. PATH = /path/to/myproject_env/bin:$PATH  （前插，优先级最高）
# 3. 保存原始 PATH 到 _OLD_VIRTUAL_PATH

# 停用（deactivate）：
# 1. 恢复 PATH
# 2. 删除 VIRTUAL_ENV
# 3. 恢复 PS1 提示符
```

### 54.2.2 site-packages 的优先级

```python
import sys

# 模块搜索路径的优先级（虚拟环境激活后）：
# 1. 脚本所在目录（''）
# 2. 虚拟环境的 site-packages
# 3. PYTHONPATH 环境变量
# 4. 系统 site-packages（如果 include-system-site-packages=true）
# 5. 标准库

# 虚拟环境的 site-packages 优先于系统
# 这确保虚拟环境中的包覆盖系统包

# 查看 site-packages 路径
import site
print(site.getsitepackages())

# .pth 文件机制
# site-packages 中的 .pth 文件会被 site 模块处理
# 每行一个路径，自动添加到 sys.path
# mypackage.pth:
# /opt/mypackage/lib
```

## 54.5 常见陷阱与最佳实践

| 陷阱 | 最佳实践 |
|------|---------|
| 忘记激活虚拟环境 | 使用 direnv 或 shell 钩子自动激活 |
| 全局安装包 | 每个项目独立虚拟环境 |
| 依赖版本不锁定 | 使用 lock 文件精确锁定 |
| .env 文件提交到 Git | .gitignore 排除环境文件 |
| 多 Python 版本需求 | pyenv + tox 或 uv |
