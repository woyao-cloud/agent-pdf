# 第八章：React 性能问题与优化策略

性能优化是 React 开发中永恒的话题。随着应用规模的增长，即使是优雅的组件架构也可能面临性能瓶颈。本章将系统性地分析 React 中的常见性能问题，并提供经过验证的优化策略和解决方案。

## 8.1 理解 React 的渲染机制

在深入优化之前，我们必须理解 React 的渲染机制。这是所有性能优化的理论基础。

### 8.1.1 渲染阶段 vs 提交阶段

React 的每次更新分为两个阶段：

**渲染阶段（Render Phase）**：
- React 调用组件函数，生成新的虚拟 DOM 树（VNode Tree）
- 这个过程是纯计算，没有副作用
- 可能被 React 中断（在并发模式下）
- 此阶段的代码运行在内存中，不影响实际 DOM

**提交阶段（Commit Phase）**：
- React 将虚拟 DOM 的差异应用到真实 DOM
- DOM 操作发生在此阶段
- `useLayoutEffect` 在此阶段同步执行
- `useEffect` 在浏览器绘制后异步执行

### 8.1.2 触发重新渲染的条件

React 组件在以下三种情况下会重新渲染：

1. **状态变化**：组件内部调用 `setState` 或 `dispatch`
2. **父组件重新渲染**：父组件重新渲染会导致所有子组件重新渲染（除非子组件被记忆化）
3. **Context 变化**：Context 值变化会导致所有消费该 Context 的组件重新渲染

### 8.1.3 Reconciliation 过程

React 的协调（Reconciliation）过程决定了哪些 DOM 节点需要更新：

1. 对比新旧两棵虚拟 DOM 树
2. 使用 `key` 属性来识别同一层级的节点
3. 只更新发生变化的节点
4. 差异计算的结果会以"副作用"的形式标记在 Fiber 节点上

理解这些基础概念后，我们来看看实际开发中最常见的性能问题。

## 8.2 常见性能问题及解决方案

### 8.2.1 不必要的重新渲染

**问题描述**：父组件重新渲染时，没有变化的子组件也跟着重新渲染。

这是最普遍的性能问题。在大型组件树中，一个状态更新可能引发成百上千个组件的重新渲染。

**解决方案 1：React.memo**

```tsx
import { memo, useState } from "react";

// 使用 memo 包裹：仅当 props 变化时才重新渲染
const ExpensiveChart = memo(function ExpensiveChart({ data }: { data: number[] }) {
  console.log("图表组件渲染"); // 仅在 data 变化时打印
  return (
    <div className="chart">
      {data.map((value, index) => (
        <div
          key={index}
          className="bar"
          style={{ height: `${value}px` }}
        />
      ))}
    </div>
  );
});

function Dashboard() {
  const [search, setSearch] = useState("");
  const [chartData, setChartData] = useState([30, 50, 80, 45, 90]);

  return (
    <div>
      <input
        value={search}
        onChange={(e) => setSearch(e.target.value)}
        placeholder="搜索..."
      />
      {/* 即使 search 变化，ExpensiveChart 也不会重新渲染 */}
      <ExpensiveChart data={chartData} />
    </div>
  );
}
```

**解决方案 2：useMemo 和 useCallback**

```tsx
import { useState, useMemo, useCallback, memo } from "react";

interface Product {
  id: number;
  name: string;
  price: number;
}

const ProductList = memo(function ProductList({
  products,
  onAddToCart,
}: {
  products: Product[];
  onAddToCart: (id: number) => void;
}) {
  console.log("ProductList 重新渲染");
  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>
          {product.name} - ¥{product.price}
          <button onClick={() => onAddToCart(product.id)}>
            加入购物车
          </button>
        </li>
      ))}
    </ul>
  );
});

function Shop() {
  const [search, setSearch] = useState("");
  const [cart, setCart] = useState<number[]>([]);
  const allProducts: Product[] = [
    { id: 1, name: "React 实战", price: 79 },
    { id: 2, name: "TypeScript 入门", price: 59 },
    { id: 3, name: "Node.js 后端开发", price: 69 },
  ];

  // useMemo：仅在依赖变化时重新计算
  const filteredProducts = useMemo(
    () =>
      allProducts.filter((p) =>
        p.name.toLowerCase().includes(search.toLowerCase())
      ),
    [search]
  );

  // useCallback：返回稳定的函数引用
  const handleAddToCart = useCallback((id: number) => {
    setCart((prev) => [...prev, id]);
  }, []);

  return (
    <div>
      <input
        value={search}
        onChange={(e) => setSearch(e.target.value)}
        placeholder="搜索商品..."
      />
      <p>购物车 ({cart.length} 件商品)</p>
      <ProductList
        products={filteredProducts}
        onAddToCart={handleAddToCart}
      />
    </div>
  );
}
```

有时最好的优化不是记忆化，而是将状态下移到更合适的层级：

### 8.2.2 大型列表重新渲染

**问题描述**：列表中的任何一个项发生变化，整个列表都重新渲染。

**解决方案：虚拟化（Virtualization）**

虚拟化的核心思想是：只渲染可视区域内的列表项，而不是渲染整个列表。

```tsx
import { useState, useRef, useCallback } from "react";
import { FixedSizeList } from "react-window";

// 使用 react-window 实现虚拟列表
function VirtualizedList({ items }: { items: string[] }) {
  // Row 组件：只渲染当前可见的项
  const Row = useCallback(
    ({ index, style }: { index: number; style: React.CSSProperties }) => (
      <div style={style} className="list-item">
        {index + 1}. {items[index]}
      </div>
    ),
    [items]
  );

  return (
    <FixedSizeList
      height={400}        // 容器高度
      width="100%"         // 容器宽度
      itemCount={items.length} // 总项数
      itemSize={35}        // 每项高度
    >
      {Row}
    </FixedSizeList>
  );
}

// 使用示例
function LargeListExample() {
  const items = Array.from(
    { length: 10000 },
    (_, i) => `列表项 ${i + 1}`
  );

  return (
    <div>
      <h2>大型列表（10000 项）</h2>
      <p>使用虚拟化：DOM 中始终只有约 12 个节点</p>
      <VirtualizedList items={items} />
    </div>
  );
}
```

**不稳定的 key 导致的性能问题**：

```tsx
// 错误：使用 index 作为 key
function BadKeyList({ items }: { items: string[] }) {
  return (
    <ul>
      {items.map((item, index) => (

  return (

        <li key={item.id}>{item.text}</li> // React 可以精准追踪每个元素
      ))}
    </ul>

}
```

### 8.2.3 昂贵的重复计算

**问题描述**：每次渲染都执行耗时的计算，即使输入没有变化。

**解决方案：useMemo**

```tsx
import { useMemo, useState } from "react";

function AnalyticsDashboard({ transactions }: { transactions: Transaction[] }) {
  // 不优化：每次渲染都重新计算
  // const totalRevenue = transactions
  //   .filter(t => t.type === 'revenue')
  //   .reduce((sum, t) => sum + t.amount, 0);

  // 优化：仅当 transactions 变化时重新计算
  const totalRevenue = useMemo(
    () =>
      transactions
        .filter((t) => t.type === "revenue")
        .reduce((sum, t) => sum + t.amount, 0),
    [transactions]
  );

  // 更复杂的聚合计算
  const monthlyStats = useMemo(
    () => computeMonthlyStats(transactions),
    [transactions]
  );

  // 排序后的数据
  const sortedTransactions = useMemo(
    () => [...transactions].sort((a, b) => b.amount - a.amount),
    [transactions]
  );

  return (
    <div>
      <h2>总收入: ¥{totalRevenue.toLocaleString()}</h2>
      <MonthlyChart data={monthlyStats} />
      <TransactionTable data={sortedTransactions} />
    </div>
  );
}

// 对于极其耗时的计算，可以考虑使用 Web Worker
function useWebWorker<T>(workerFn: () => T, inputs: unknown[]) {
  const [result, setResult] = useState<T | null>(null);

  useEffect(() => {
    const worker = new Worker(
      URL.createObjectURL(
        new Blob([
          `self.onmessage = function() {
            const result = (${workerFn.toString()})();
            self.postMessage(result);
          }`,
        ])
      )
    );

    worker.onmessage = (e) => setResult(e.data);
    worker.postMessage(null);

    return () => worker.terminate();
  }, inputs);

  return result;
}
```

### 8.2.4 Context 引起的广泛重新渲染

**问题描述**：Context 值变化时，所有消费该 Context 的组件都会重新渲染，即使它们只使用了未变化的部分。

**解决方案 1：拆分 Context**

```tsx
import { createContext, useContext, useState, memo } from "react";

// 错误做法：所有状态放在一个 Context 中
const AppContext = createContext<{
  user: User | null;
  theme: "light" | "dark";
  notifications: Notification[];
  // ...更多状态
} | null>(null);

// 优化：按职责拆分 Context
const UserContext = createContext<User | null>(null);
const ThemeContext = createContext<"light" | "dark">("light");
const NotificationContext = createContext<Notification[]>([]);

// 各个组件只订阅自己需要的 Context
const UserAvatar = memo(function UserAvatar() {
  const user = useContext(UserContext);
  return <img src={user?.avatar} alt={user?.name} />;
});

const ThemeToggle = memo(function ThemeToggle() {
  const theme = useContext(ThemeContext);
  return <button>{theme === "dark" ? "☀️" : "🌙"}</button>;
});

const NotificationBell = memo(function NotificationBell() {
  const notifications = useContext(NotificationContext);
  return <span>通知 ({notifications.length})</span>;
});
```

**解决方案 2：Context 值记忆化**

```tsx
function AppProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null);
  const [theme, setTheme] = useState<"light" | "dark">("light");

  // 关键：使用 useMemo 确保 context value 的引用稳定性
  const userValue = useMemo(() => ({ user, setUser }), [user]);
  const themeValue = useMemo(() => ({ theme, setTheme }), [theme]);

  return (
    <UserContext.Provider value={userValue}>
      <ThemeContext.Provider value={themeValue}>
        {children}
      </ThemeContext.Provider>
    </UserContext.Provider>
  );
}
```

**解决方案 3：使用状态管理库**

对于全局状态，考虑使用 Zustand、Jotai 或 Recoil 等状态管理库，它们内置了细粒度的订阅机制：

```tsx
import { create } from "zustand";

// Zustand store：组件只在使用到的状态变化时重新渲染
const useStore = create<{
  user: User | null;
  theme: "light" | "dark";
  notifications: Notification[];
  setUser: (user: User) => void;
  setTheme: (theme: "light" | "dark") => void;
}>((set) => ({
  user: null,
  theme: "light",
  notifications: [],
  setUser: (user) => set({ user }),
  setTheme: (theme) => set({ theme }),
}));

// 仅订阅 user，不会因 theme 或 notifications 变化而重新渲染
function UserProfile() {
  const user = useStore((state) => state.user);
  return <div>{user?.name}</div>;
}

// 仅订阅 theme，不会因其他状态变化而重新渲染
function ThemeSwitcher() {
  const theme = useStore((state) => state.theme);
  const setTheme = useStore((state) => state.setTheme);
  return (
    <button onClick={() => setTheme(theme === "dark" ? "light" : "dark")}>
      切换主题
    </button>
  );
}
```

### 8.2.5 Props 钻取

**问题描述**：数据需要通过多层组件传递，中间组件即使不使用这些数据也要参与转发。

**解决方案 1：组合模式（Component Composition）**

```tsx
// 问题代码：props 钻取
function Page() {
  const [user] = useState<User>(/* ... */);
  return (
    <Layout>
      <Header user={user} />
      <Sidebar user={user}>
        <Navigation user={user} />
      </Sidebar>
    </Layout>
  );
}

// 优化：使用组合模式
function OptimizedPage() {
  const [user] = useState<User>(/* ... */);

  return (
    <Layout>
      {/* 直接在 Page 中渲染需要 user 的组件 */}
      <Header>
        <UserAvatar user={user} />
      </Header>
      <Sidebar>
        {/* 通过 children 传递，Sidebar 不需要知道 user */}
        <Navigation>
          <UserMenu user={user} />
        </Navigation>
      </Sidebar>

}

// 通用布局组件，不关心具体内容
function Sidebar({ children }: { children: React.ReactNode }) {
  return (
    <aside className="sidebar">
      <div className="sidebar-content">{children}</div>
    </aside>
  );
}

function Header({ children }: { children: React.ReactNode }) {
  return (
    <header className="header">
      <div className="header-left">{children}</div>
    </header>
  );
}
```

### 8.2.6 无限的 useEffect 循环

**问题描述**：useEffect 的依赖项在每次渲染后都发生变化，导致无限循环。

**常见原因及解决方案**：

```tsx
// 问题 1：对象/数组作为依赖
function Problem1({ data }: { data: Record<string, number> }) {
  const [processed, setProcessed] = useState<number[]>([]);

  // 错误：每次渲染 data 都是新对象引用
  useEffect(() => {
    setProcessed(Object.values(data));
  }, [data]); // 每次都触发

  // 解决方案：使用具体值而非对象引用
  useEffect(() => {
    setProcessed(Object.values(data));
  }, [JSON.stringify(data)]); // 或使用深比较库
}

// 问题 2：函数作为依赖
function Problem2() {
  const [count, setCount] = useState(0);

  // 错误：每次渲染 handleClick 都是新函数
  // const handleClick = () => setCount((c) => c + 1);

  // 解决方案 1：使用 useCallback
  const handleClick = useCallback(() => {
    setCount((c) => c + 1);
  }, []);

  useEffect(() => {
    window.addEventListener("click", handleClick);
    return () => window.removeEventListener("click", handleClick);
  }, [handleClick]);

  return <div>Count: {count}</div>;
}

// 问题 3：在 effect 中更新引发依赖变化的状态
function Problem3() {
  const [items, setItems] = useState<number[]>([]);

  // 错误：setItems 触发重新渲染，items 变化，effect 再次执行
  useEffect(() => {
    fetch("/api/items")
      .then((res) => res.json())
      .then((data) => setItems(data.items));
  }, [items]); // 去掉 items 依赖

  // 正确：使用空的依赖数组
  useEffect(() => {
    fetch("/api/items")
      .then((res) => res.json())
      .then((data) => setItems(data.items));
  }, []); // 只在挂载时执行
}

// 问题 4：useRef 作为稳定引用
function Problem4() {
  const [count, setCount] = useState(0);
  const latestCount = useRef(count);

  // 同步 ref
  useEffect(() => {
    latestCount.current = count;
  });

  // 使用 ref 避免闭包陷阱
  const handleClick = useCallback(() => {
    console.log("当前计数:", latestCount.current);
    setCount((c) => c + 1);
  }, []);

  return <button onClick={handleClick}>点击</button>;
}
```

### 8.2.7 布局抖动（Layout Thrashing）

**问题描述**：在同一个渲染帧中反复读取然后写入 DOM 样式，强制浏览器进行多次布局计算。

```tsx
import { useLayoutEffect, useRef } from "react";

// 问题代码：布局抖动
function LayoutThrashing() {
  const listRef = useRef<HTMLUListElement>(null);

  useEffect(() => {
    const items = listRef.current?.children;
    if (!items) return;

    for (let i = 0; i < items.length; i++) {
      // 读取：强制浏览器计算布局
      const width = items[i].clientWidth;
      // 写入：使之前的布局失效
      (items[i] as HTMLElement).style.width = `${width * 1.5}px`;
      // 循环中反复读取/写入，造成严重的布局抖动
    }
  });

  return (
    <ul ref={listRef}>
      {Array.from({ length: 100 }, (_, i) => (
        <li key={i}>项目 {i}</li>
      ))}
    </ul>
  );
}

// 优化：批量读取，批量写入
function OptimizedLayout() {
  const listRef = useRef<HTMLUListElement>(null);

  useLayoutEffect(() => {
    const items = listRef.current?.children;
    if (!items) return;

    // 第一步：批量读取所有尺寸
    const widths: number[] = [];
    for (let i = 0; i < items.length; i++) {
      widths.push(items[i].clientWidth);
    }

    // 第二步：批量写入所有样式
    for (let i = 0; i < items.length; i++) {
      (items[i] as HTMLElement).style.width = `${widths[i] * 1.5}px`;
    }
  });

  return (
    <ul ref={listRef}>
      {Array.from({ length: 100 }, (_, i) => (
        <li key={i}>项目 {i}</li>
      ))}
    </ul>
  );
}
```

### 8.3.1 React DevTools Profiler

React DevTools 的 Profiler 是性能分析的首选工具：


### 8.3.2 useWhyDidYouUpdate 模式

一个实用的调试 Hook，帮助找出为什么组件重新渲染：

```tsx
import { useEffect, useRef } from "react";

function useWhyDidYouUpdate(
  componentName: string,
  props: Record<string, unknown>
) {
  const previousProps = useRef<Record<string, unknown>>({});

  useEffect(() => {
    const changes: Record<string, { from: unknown; to: unknown }> = {};

    // 合并所有 key
    const allKeys = Object.keys({
      ...previousProps.current,
      ...props,
    });

    allKeys.forEach((key) => {
      if (previousProps.current[key] !== props[key]) {
        changes[key] = {
          from: previousProps.current[key],
          to: props[key],
        };
      }
    });

    if (Object.keys(changes).length > 0) {
      console.group(`[为什么重新渲染] ${componentName}`);
      console.table(changes);
      console.groupEnd();
    }

    previousProps.current = props;
  });
}

// 使用示例
const ExpensiveComponent = memo(function ExpensiveComponent(props: {
  data: number[];
  label: string;
}) {
  useWhyDidYouUpdate("ExpensiveComponent", props);

  return (
    <div>
      <h3>{props.label}</h3>
      <p>数据长度: {props.data.length}</p>
    </div>
  );
});
```

### 8.3.3 React.memo 自定义比较函数

```tsx
interface User {
  id: number;
  name: string;
  email: string;
  lastLogin: Date;
}

const UserCard = memo(
  function UserCard({ user }: { user: User }) {
    return (
      <div className="user-card">
        <h3>{user.name}</h3>
        <p>{user.email}</p>
      </div>
    );
  },
  (prevProps, nextProps) => {
    // 自定义比较：只比较 id，忽略其他字段
    return prevProps.user.id === nextProps.user.id;
  }
);
```

## 8.5 性能优化的黄金法则

1. **先测量，再优化**：在没有 Profiler 数据之前，不要猜测性能瓶颈。人类直觉在性能问题上往往是错误的。

2. **关注关键路径**：优先优化首屏加载和核心交互路径上的性能问题。次要页面的 100ms 优化不如核心路径的 10ms 优化有价值。

3. **避免过早优化**：在遇到实际的性能问题之前，优先保证代码的可读性和可维护性。使用 React.memo 等优化手段需要衡量其本身的成本。

4. **渐进式优化**：不要试图一次性优化所有内容。找到最大的瓶颈，解决它，测量效果，然后继续下一个。

5. **理解框架的取舍**：React 的虚拟 DOM 和协调机制本身就是为了平衡性能和开发体验。在某些极端场景下，直接操作 DOM 可能更快，但这通常不值得牺牲声明式编程的优势。
